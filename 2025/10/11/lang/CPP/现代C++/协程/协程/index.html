协程

ZaynPei Lv6
  2025-10-11 17:49:06 2025-10-11 17:49:06 Created   2025-10-18 08:30:09 2025-10-18 08:30:09 Updated  4.4k Words  15 Mins

协程, 也可以叫做轻量级的线程或者用户线程, 它是一种执行过程中能够yield(暂停)和resume(恢复)的子程序.

想象一下我们平时写的普通函数（Function/Subroutine）： - 入口只有一个：从函数的第一行代码开始执行。 - 出口只有一个：执行到 return 语句或函数末尾就结束了。 - 中途不记事：一旦函数返回，它内部的所有局部变量都会被销毁，下次再调用，一切都是从头开始。

现在，想象一个“超级函数”，它拥有以下特异功能： - 可以在执行到一半时主动暂停，并把控制权交还给调用者。 - 当它暂停时，会完整地保存自己当前的所有状态（比如局部变量的值、代码执行到哪一行了）。 - 下次调用者想让它继续时，它可以从上次暂停的地方无缝恢复，继续执行。

这个“超级函数”，就是我们要提到的协程

协程的核心思想是在单个线程内实现多任务切换，通过在用户态保存和恢复执行上下文来实现任务间的切换，而不需要操作系统内核的参与。这样可以大幅减少上下文切换的开销，提高并发性能。

  • 协作式调度 (Cooperative Scheduling): 这是协程与线程最根本的区别。线程的切换是由操作系统内核决定的（抢占式），比如一个线程的时间片用完了，操作系统会强制把它换下来。而协程的切换是由程序自己控制的。一个协程只有在它自己主动调用 yield (或者 await, swapcontext 等) 时，才会放弃CPU控制权。

  • 状态保持 (State Preservation): 当协程暂停时，它的整个调用栈（包括局部变量、参数、程序计数器等）都会被保存下来。当它恢复时，这些状态会被原封不动地还原。

  • 在单线程中运行 (Runs in a Single Thread): 多个协程可以在一个线程中“轮流”执行。这带来了巨大的好处：因为它们不涉及真正的并行，所以访问共享数据时几乎不需要加锁（只要切换点是明确的），极大地简化了编程模型，避免了多线程编程中常见的竞态条件和死锁问题。

主要优势

  • 极高的并发能力: 操作系统的线程是重量级资源，创建一个线程需要进入内核，分配几MB的栈内存。一个系统能同时运行的线程数量通常是几百到几千。而协程是用户态的轻量级资源，创建和切换的成本极低（接近于一次函数调用），栈内存通常也只需几KB。一个线程可以轻松管理数万甚至数百万个协程。

  • 极低的切换开销 (Context Switch Cost): 线程切换需要从用户态陷入内核态，保存和恢复大量的寄存器和内核数据结构，成本较高。而协程切换完全在用户态发生，只需保存/恢复少量关键寄存器，速度极快。

  • 用同步的方式编写异步代码: 在协程出现之前，处理高并发I/O通常使用回调函数（Callback），容易形成“回调地狱”，代码可读性和可维护性极差。使用协程，我们可以这样写代码：

目前, 协程可以通过多种方式实现：

  • 底层库支持：如 Linux 的 ucontext 系列函数，提供完整的上下文保存和恢复能力。
  • 汇编：直接操作寄存器，实现最极致的性能，如 Boost.Context。
  • 语言级支持：这是目前最主流的方式。语言内置 async/await、go、yield 等关键字，让编译器和运行时来处理复杂的上下文切换。
    • Go: goroutine 是语言的招牌特性。
    • Python: asyncio 库和 async/await 语法。
    • C++20: 引入了标准的协程支持 (co_await, co_yield)。

对称协程与非对称协程

这两种协程的主要区别在于切换控制权的方式。

非对称协程 (Asymmetric Coroutines): 这是我们最常见到的协程模型，可以理解为一种主从关系或调用者/被调用者关系。

这种协程存在一个明确的“调用者”（通常是主函数或调度器）和一个或多个“被调用者”（协程）。控制权总是在调用者和一个协程之间来回传递, 协程不能将控制权直接交给另一个协程，它只能暂停并把控制权返回给它的调用者。

关键操作: - resume ：调用者启动或恢复一个协程。 - yield ：协程暂停执行，并将控制权和可能的返回值交还给调用者。

对称协程 (Symmetric Coroutines): 所有协程都是平等的对等体 (Peers), 这种模型更加灵活和强大

任何一个协程都可以将控制权直接转移给任何其他一个协程，而不需要经过一个中央调度器或调用者。协程之间没有调用者/被调用者的等级之分。

其只有一个核心操作，通常称为 transfer (或 yield to)。 - transfer(target_coroutine)：当前协程暂停，并立即将CPU控制权交给 target_coroutine。 - 我们之前了解过的 ucontext 中的 swapcontext(current_ctx, target_ctx) 就是一个 transfer 操作。

有栈协程与无栈协程

这个分类的核心在于：协程暂停时，它的状态（特别是函数调用栈）是如何保存的？

有栈协程 (Stackful Coroutines): 每个协程都拥有一个独立的、完整的执行栈（一块预先分配的内存，通常较大，如几KB到几MB）。当协程切换时，本质上是切换CPU的栈指针（SP寄存器）指向不同的协程栈。因为保存了完整的调用栈，所以协程可以在任意深的嵌套函数调用中暂停。

这种模型非常强大，协程可以像普通函数一样调用其他函数，甚至递归调用自己, 程序员在编写函数时完全不需要手动创建协程或者声明关键字, 这使得协程的使用更加自然和简洁。缺点是每个协程都需要分配一块栈内存，数量多时会占用较多内存。

Go 语言的 goroutine 就是典型的有栈协程。每个 goroutine 初始栈大小只有几KB，但可以根据需要动态增长。Go 的调度器会在多个 goroutine 之间高效地切换。

在执行时, Go语言的运行时调度器 (Runtime Scheduler)在背后做了什么：

  • 函数劫持：调用的 connection.Read 并不是一个天真的、直接的系统调用。它是Go运行时“动过手脚”的版本。
  • 发起非阻塞I/O：这个函数会向操作系统发起一个非阻塞的读请求，并告诉操作系统：“数据准备好后，请通知我的调度器。”
  • 主动让出：然后，这个函数会调用运行时调度器的API，主动告诉调度器：“我这个goroutine（协程）现在要等待数据，请把我暂停，然后去运行队列里的其他goroutine吧。”
  • 上下文切换：调度器接收到通知，保存当前goroutine的整个栈，将其标记为“等待中”，然后从队列里取出另一个“可运行”的goroutine，恢复它的栈，让它继续在同一个OS线程上执行。

也就是, 在运行时这个调度器把阻塞操作的复杂性完全封装起来了，让程序员感觉不到它的存在。

无栈协程 (Stackless Coroutines): 协程不拥有独立的执行栈，它们都在同一个共享的系统栈上运行。

当一个协程暂停时，它实际上是像普通函数一样 return 了。编译器会在编译时进行“状态机转换”，将协程函数转换成一个对象。这个对象会保存所有局部变量的值和当前执行的位置（状态）。当协程恢复时，实际上是重新调用这个函数，函数根据保存的状态跳转到正确的位置继续执行。

这种模型的优点是内存开销极小，因为不需要为每个协程分配独立的栈空间, 只需在堆上分配一个很小的状态对象，可以创建海量的协程。。缺点是协程只能在函数的顶层暂停，不能在嵌套调用中间暂停。C++20 的协程 (co_await)就是典型的无栈协程。

对于无栈协程, 你必须在函数签名上使用关键字 (如 co_await, async) 来标记这个函数是一个协程。使得在编译阶段, 编译器自动把这个函数转换成一个状态机。

嘿，编译器，我正在写的这个函数不是一个普通的函数。看到 co_await 这个标记了吗？这代表这里是一个潜在的暂停点。请你不要为我生成普通的函数代码，而是把它转换成一个状态机。”

编译器做了什么：

  • 识别标记：编译器扫描代码，co_await 就是它寻找的信号。
  • 转换代码：它将你的协程函数大卸八块，变成一个状态机对象。函数里的每个 co_await 点，都成为状态机的一个状态转换点。
  • 保存状态：它分析出哪些局部变量在 co_await 之后还需要被使用，然后把这些变量从函数栈上“搬家”到堆上分配的协程帧对象里。
  • 生成返回逻辑：在 co_await 的位置，编译器会插入代码，用于保存当前状态、然后 return 控制权给调用者（事件循环）。

因此, 如果没有 co_await 这个明确的标记，编译器就完全不知道哪里需要暂停，它就会把你的函数当作一个普通的、同步的函数来编译。在这种情况下, 这个函数一旦遇到真正的阻塞I/O就会阻塞整个线程。

What Color is Your Function?

2015年，程序员 Bob Nystrom 在博客上发表文章 [What Color is Your Function], 以设计编程语言的角度，探讨同步函数与异步函数、以及他们在语言层的最优实现。

想象一下，你的编程语言里有两种颜色的函数：

  • 蓝色函数 (Blue Functions)：这些是普通的、同步的 C++ 函数。你调用它，它会执行到底，然后返回一个结果。简单直接。
  • 红色函数 (Red Functions)：这些是异步的协程函数。你调用它，它会立即返回一个“承诺”（Promise/Future），而不是最终的结果。要获得最终结果，你必须 await 这个“承诺”。

现在，这个世界有一条非常严格且显而易见的规则：红色函数可以调用蓝色函数，但蓝色函数不能调用红色函数。

为什么？因为蓝色函数（同步）的编写者期望当他调用一个函数时，能立刻得到一个具体的值（比如 int 或 std::string）。但他调用红色函数时，得到的却是一个“承诺”对象。蓝色函数完全不知道该如何处理这个“承诺”，它没有 await 的能力，它无法暂停自己去等待这个承诺被兑现。它会卡住，不知所措。为了能处理这个“承诺”，调用者自己也必须有 await 的能力。而拥有 await（在 C++ 中是 co_await）能力的函数，本身就必须是一个红色函数（协程）。

反思一下, 这个问题的实际后果是什么？

  • 割裂的生态系统：你会发现库被分成了两个世界。“同步库”（比如一个同步的HTTP客户端）和“异步库”（一个异步的HTTP客户端）。在一个红色函数里，你如果调用了一个同步库的阻塞函数，会阻塞整个线程，让你所有的并发优势荡然无存。

  • 调用栈的传染：如上例所示，一旦你决定在某个地方使用异步，这种异步性会不可避免地蔓延到所有调用它的地方。最终，这个红色调用链会到达最顶端，通常是 main 函数（一个蓝色函数）。你必须在 main 函数里使用一些特殊的“桥接”代码来启动并等待最顶层的红色函数完成。这通常是一个阻塞的 .get() 调用，或者一个事件循环的 run() 方法。

调用者完全不需要知道它调用的函数是阻塞的还是非阻塞的。这就是有栈协程消除了“函数染色”问题的方法。实际上, 所有函数都是“红色”的，因为它们都可能在某个时刻被暂停和恢复，但对程序员来说，它们看起来都是普通的蓝色函数。

总结一下，有栈协程和无栈协程的主要区别：

特性 无栈协程 (C++20) 有栈协程 (Go)
谁负责暂停 编译器（根据 co_await 标记） 运行时（在 I/O 函数内部）
暂停点 显式（co_await 所在位置） 隐式（在标准库的 I/O 调用中）
需要关键字吗 必须需要（co_await），否则无法工作 完全不需要，对程序员透明
函数颜色 有颜色（“红色”函数必须被 await） 无颜色（所有函数看起来都一样）
实现代价 编译期转换，几乎零运行时开销，内存极小 强大的运行时调度器，内存开销大（独立栈）
