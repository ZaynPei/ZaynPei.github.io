多线程模型

首先, 我们需要了解两种基本的线程类型：

  • 用户线程 (User Threads)：由用户空间的线程库（比如一个语言的运行时）来管理（例如 Pthreads, Java threads）。操作系统内核对这些线程一无所知，它只看到了一个单线程进程。
    • 核心意义：轻量级与高效切换
      • 创建与销毁快：创建线程不需要经过操作系统，只是在用户空间分配一些数据结构。
      • 切换（上下文切换）极快：线程A切换到线程B，只是库函数的一次调用，保存一下CPU寄存器，然后跳转到线程B的执行点。这个过程不需要陷入内核 (System Call)，开销非常小。
    • 重大局限：
      • 阻塞问题：由于内核“看”不到这些用户线程，它只“看”到这一个进程（或者说，承载这些用户线程的那个内核线程）。如果其中任何一个用户线程发起了一个阻塞式的系统调用（例如 read 等待磁盘I/O），内核会把整个进程（或承载线程）挂起。这意味着，该进程内的所有其他用户线程，即使它们是就绪的，也无法运行。
      • 无法利用多核：内核只知道一个执行实体，它只会把这个进程调度到一个CPU核心上。你无法通过纯粹的用户线程实现真正的并行（Parallelism），只能实现并发（Concurrency）。
  • 内核线程 (Kernel Threads)：由操作系统内核直接支持和管理。几乎所有现代操作系统（Windows, Solaris, Linux, macOS）都实现了内核线程。
    • 核心意义：真正的并发与健壮性
      • 内核调度：内核“看得到”每一个线程。如果线程A因为I/O阻塞了，内核知道它被阻塞了，会立刻调度同一个进程中的线程B（如果B是就绪态）去运行。
      • 多核并行：内核调度器可以将同一个进程的多个内核线程，同时调度到不同的CPU核心上运行，实现真正的并行计算。
    • 相应代价: 开销更高. 线程的创建、销毁和切换（上下文切换）都必须陷入内核来完成。这个模式切换（User Mode <-> Kernel Mode）的开销远大于用户空间的函数调用。

而多线程模型就是描述如何将用户线程映射到内核线程的三种不同方式。

多对一 (Many-to-One) 模型

将许多用户级线程映射到一个内核级线程上 。

  • 线程管理（创建、销毁、调度）完全在用户空间由线程库完成。
  • 操作系统内核根本不知道这些用户线程的存在，它只认为这是一个普通的单线程进程 。

优点： - 高效：由于线程管理不需要进入内核，没有系统调用的开销，因此速度非常快 。

缺点： - 阻塞问题：最大的问题是，如果任何一个用户线程执行了阻塞的系统调用（例如读取文件），那么整个进程（包括所有其他用户线程）都会被阻塞 。 - 无法并行：由于在任何时刻只有一个内核线程在运行，因此即使在多核处理器上，该进程也无法实现真正的并行。

一对一 (One-to-One) 模型

将每一个用户级线程都映射到一个单独的内核级线程 。

  • 每当创建一个用户线程，线程库就会相应地请求内核创建一个内核线程。
  • 线程的调度和管理主要由内核完成。

优点： - 高并发性：一个线程的阻塞不会影响其他线程 。 - 支持并行：由于有多个内核线程，该进程可以在多核处理器上实现真正的并行执行。

缺点： - 开销大：每创建一个用户线程都需要创建一个对应的内核线程，这会带来显著的开销和资源占用 。系统能支持的内核线程数量是有限的。

主流应用：这是目前最主流的模型，被 Windows、Linux 和现代 Solaris 等操作系统采用 。

多对多 (Many-to-Many) 模型

将多个用户级线程复用到多个内核级线程上 。

  • 这是一种混合模型。内核线程的数量可以少于或等于用户线程的数量。
  • 用户空间的线程库负责将“准备就绪”的用户线程调度到可用的内核线程上去执行 。

优点：

  • 灵活性高：结合了前两种模型的优点 。
  • 解决阻塞问题：一个用户线程阻塞时，内核可以调度同一个进程中的另一个用户线程（通过另一个内核线程）来执行。
  • 开销适中：允许程序创建任意多的用户线程，而内核线程的数量则可以由操作系统根据负载动态调整 。

缺点：实现起来非常复杂。

变体：二级 (Two-level) 模型

定义：这本质上是“多对多”模型的一个变体，它额外允许一个用户线程绑定 (bound)到某一个特定的内核线程上 。

工作方式：大部分用户线程在多个内核线程池中灵活调度（多对多），而个别需要高优先级或实时响应的线程则可以“霸占”一个内核线程（一对一）。

优点：提供了极高的灵活性，兼顾了高效调度和实时响应的需求。

实际应用

在现代主流操作系统中（如 Linux, Windows, macOS），我们平时使用的线程模型绝大多数是 1:1 模型。

它意味着你（用户）在代码中每创建一个线程，操作系统内核就会为你创建一个对应的内核线程。

你用来创建线程的库（如 C++ 的 std::thread，C 的 pthread，Python 的 threading.Thread，Java 的 Thread）实际上只是一个“包装器”（Wrapper）。

当你调用 std::thread myThread(func) 时，std::thread 的构造函数内部会调用系统API（例如 Linux 上的 clone() 系统调用）来请求内核创建一个内核线程。

我们享受着内核线程的所有好处：一个线程阻塞I/O，不影响其他线程；多线程可以被自动调度到多核上并行运行。

我们也承受着它的代价：创建和切换线程是有固定开销的。因此，我们不能无限制地创建成千上万个线程（这会导致资源耗尽和调度开销过大）。

不过, “用户线程”的核心思想（即在用户空间实现极轻量级的切换）并没有消失，它以一种新的形式回归了，这就是我们常说的协程 (Coroutines)。

协程是一种 M:N 模型, 它结合了用户线程和内核线程的优点。

  • 系统创建 N 个内核线程（通常等于CPU核心数，组成一个线程池）。
  • 语言运行时（Runtime，如 Go 的调度器、Python 的 asyncio 事件循环）在用户空间创建 M 个协程（M 可以非常大，如成千上万个，M >> N）。
  • 运行时负责将这 M 个协程“投放”到 N 个内核线程上执行。

这就是现代版的“用户线程”：Go 语言的 Goroutine 是最典型的例子。你可以轻松创建几十万个 Goroutine。Go 的调度器会在用户态管理它们，并将它们调度到少量的OS线程（内核线程）上。

当一个 Goroutine 遇到I/O（Go的运行时会使用非阻塞I/O），Go调度器会自动将其“挂起”，并让这个OS线程去执行另一个就绪的 Goroutine。

Python 的 asyncio、C# 的 async/await、Rust 的 tokio 也是同理。它们都在一个（或少数几个）内核线程上，通过事件循环在用户态管理着大量的“任务”（Tasks）或“协程”。

一些多线程问题

fork() 和 exec() 系统调用的语义

当一个多线程进程中的某个线程调用 fork() 来创建新进程时，会产生一个歧义：新的子进程是应该复制父进程的所有线程，还是只复制那个调用 fork() 的线程？

  • 如果子进程复制了所有线程，那么它也必须复制所有线程的锁、状态等，这非常复杂且容易出错。

  • 如果子进程只复制了调用 fork() 的线程，那么其他线程（例如可能持有锁的线程）就消失了，这可能导致子进程立即陷入死锁。

解决方案是：一些 UNIX 系统提供了两个不同版本的 fork() ，一个只复制调用者线程，另一个复制所有线程，让程序员根据需求选择。

不过 POSIX 标准倒是规定：fork() 会创建一个新的子进程，这个子进程的内存空间是父进程的完整副本，但它的执行实体（线程）只有一个，即那个调用了 fork() 的线程的副本。

但是在大部分情况下，fork()与exec()会配合使用, 因此带来的锁问题根本不重要。因为子进程在有机会再次使用那个有问题的锁之前，它的整个内存就被 exec() 覆盖了。

不过与 fork() 不同，exec() 的语义是清晰的 。当任何线程调用 exec() 时，它会用一个新程序替换整个进程，包括该进程中的所有线程和资源。

线程取消 (Thread Cancellation)

线程取消是指一个线程请求终止另一个线程的执行。这在多线程编程中是一个复杂的问题，因为强制终止一个线程可能会导致资源泄漏、不一致状态等问题。

两种取消方式 ：

  • 异步取消 (Asynchronous cancellation)：一个线程立即终止另一个目标线程 。
    • 这种方式非常危险，因为目标线程可能正处于更新共享数据或持有锁的中间状态。强行终止它会导致数据不一致或资源无法释放。
  • 延迟取消 (Deferred cancellation)：目标线程会周期性地检查一个标志位，看自己是否应该被取消 。
    • 这是更安全的方式。线程只会在代码中预设的“安全点”（称为取消点）检查标志位。如果发现需要被取消，它可以先执行必要的清理工作（如释放锁、关闭文件），然后再安全地自行终止。

信号处理 (Signal Handling)

信号 (Signals) 是 UNIX 系统中用于通知进程发生某个事件（如非法内存访问或 Ctrl+C）的机制 。在一个多线程进程中，当一个信号到达时，应该由哪个线程来接收和处理它？

POSIX 标准规定：信号可以被传递给进程中的任何一个线程，具体取决于线程的信号掩码 (Signal Mask) 。如果一个线程没有屏蔽某个信号，那么该信号可以被传递给它。

可能的传递情况 ：

  • 将信号传递给该信号所适用的特定线程 。
  • 将信号传递给进程中的每一个线程 。
  • 将信号传递给进程中的某些特定线程 。
  • 指派一个特定的线程来接收该进程的所有信号 。

为了简化信号处理，程序员通常会创建一个专门的线程来处理信号。这个线程会阻塞所有其他信号，只等待和处理特定的信号事件 。这样可以避免多个线程同时处理同一个信号导致的竞态条件和复杂性。

线程私有数据 (Thread-Specific Data)

虽然线程的主要优点是共享数据，但有时我们希望每个线程拥有自己的私有数据副本 。例如，每个线程可能需要一个唯一的错误码变量（errno）或事务ID。

为此，许多线程库（如 Pthreads）提供了线程局部存储 (Thread-Local Storage, TLS) 机制 。程序员可以为每个线程定义私有数据，这些数据对其他线程不可见。

实现方式 ： - 使用特殊的关键字（如 C++11 的 thread_local）来声明线程局部变量。 - 使用线程库提供的 API（如 pthread_key_create, pthread_setspecific, pthread_getspecific）来管理线程私有数据。

这样，每个线程都可以安全地访问和修改自己的私有数据，而不会干扰其他线程。

具体实现

线程的实现通常涉及以下几个关键组件： - 线程控制块 (Thread Control Block, TCB)：每个线程都有一个 TCB，包含线程的状态、寄存器上下文、堆栈指针、优先级等信息。 - 线程调度器 (Thread Scheduler)：负责管理线程的创建、销毁和调度。它决定哪个线程获得 CPU 时间片。 - 同步机制 (Synchronization Mechanisms)：如互斥锁 (Mutexes)、条件变量 (Condition Variables) 等，用于协调线程之间的访问共享资源。 - 上下文切换 (Context Switching)：当线程被调度时，操作系统需要保存当前线程的状态并加载下一个线程的状态。这涉及保存和恢复寄存器、堆栈指针等信息。 - 信号处理 (Signal Handling)：线程需要能够处理异步事件，如信号。线程库通常提供机制来注册信号处理程序。 - 线程库 (Thread Libraries)：提供创建、管理和同步线程的 API，如 Pthreads、Windows 线程 API 等。

这些组件协同工作，实现了多线程的高效管理和调度，使得程序能够充分利用多核处理器的计算能力，同时保持数据的一致性和完整性。

Pthreads

Pthreads 是一个标准，而不是一个具体的实现。它为多线程编程提供了一套统一的“规则”或“接口”。

它是 POSIX 标准 (IEEE 1003.1c) 的一部分，它定义了一套用于线程创建和同步的 C 语言 API。

不过该标准只规定了 API 的行为（例如，pthread_create() 函数应该如何被调用，它会做什么），但并不规定它在操作系统内部的具体实现 。这意味着不同的操作系统可以有不同的方式来实现这套标准。

目前, Pthreads 是 UNIX 类操作系统（如 Solaris, Linux, Mac OS X）上通用的线程标准 。

Pthreads API 可以被实现为用户级库，也可以被实现为内核级库 。 在早期的某些系统中，Pthreads 可能是作为用户级库实现的（遵循“多对一”模型）。但在现代操作系统中（尤其是 Linux），Pthreads 通常是作为内核级库实现的，与“一对一”模型紧密配合，性能更高。

Linux 线程

Linux 在内核层面实现线程的方式非常独特，它并没有严格区分“进程”和“线程”。相反, Linux 有这样一个统一的核心概念：任务 (Tasks)

因此, 线程（或任务）的创建是通过 clone() 系统调用完成的。

clone() 系统调用非常灵活，它允许新创建的子任务选择性地共享其父任务的资源，例如地址空间（内存）。

  • 当调用 clone() 时，如果设置了标志位要求共享地址空间、文件描述符等资源，那么创建出的新“任务”在行为上就等同于一个线程。

  • 如果调用 clone() 时没有设置这些共享标志位（即复制所有资源），那么创建出的新“任务”在行为上就等同于一个进程（类似于 fork() 的效果）。

这种设计使得 Linux 能够非常高效地实现“一对一”线程模型 ，因为在内核看来，创建一个“线程”和一个“进程”的底层机制是统一的，只是共享的资源程度不同而已。

在 Linux 系统上，Pthreads 是程序员在应用层使用的标准 API，而内核提供的 clone() 系统调用和“任务”机制，则是实现该 API 的底层基础。
