IP协议

IP协议是网络层的核心协议，负责在不同网络之间传输数据包。在网络层中, 与IP协议密切相关的还有ICMP协议和ARP协议, 即Internet Control Message Protocol (ICMP) 和 Address Resolution Protocol (ARP)。前者用于传递控制信息和错误报告，后者用于将IP地址解析为物理地址（MAC地址）。

这里的IGMP是Internet Group Management Protocol, 用于管理主机组成员资格, 主要用于IP多播通信中, 允许主机动态加入或离开多播组。而RARP已经基本被废弃, 现在主要使用DHCP来实现类似功能。

异构网络互连

因特网并非一个单一的网络，而是由全球数以百万计的、拓扑结构、性能和协议各不相同的异构网络，通过路由器互连而成的 。要让这些千差万别的网络能够顺畅通信，就需要解决诸如不同的寻址方案、路由技术、最大分组长度等问题 。

IP协议正是解决这个问题的关键。通过让所有异构网络都使用相同的网际协议（IP），从网络层的角度看，整个因特网就好像是一个统一的、无缝的 IP网 。当主机在这个IP网上通信时，它们无需关心底层网络的具体异构细节，就好像在一个单一网络上通信一样 。

IPv4地址及其编址方法

IPv4地址是分配给IP网上每一个主机或路由器接口的一个全球唯一的32比特标识符。为了方便人类读写，32位的二进制地址通常采用 点分十进制 表示。即将32位二进制数每8位为一组，将每组转换为十进制数，并用点号.分隔。

例如：二进制 00001010 11110000 00001111 10101010 转换为点分十进制就是 10.240.15.170 。

IPv4地址的编址方法经历了三个历史阶段：分类编址 (Classful Addressing)–>划分子网 (Subnetting)–>无分类编址 (Classless Inter-Domain Routing, CIDR)

分类编址

这是最古老的编址方法，将32位IP地址分为 网络号 和 主机号 两部分。 网络号用于标识主机所在的网络，主机号用于标识网络中的具体主机。同一个网络内的所有主机具有相同的网络号, 而主机号则唯一标识该网络中的每一台主机。

分类编址将IP地址划分为A、B、C、D、E五类, 其中A、B、C类地址用于主机寻址, D类地址用于多播, E类地址保留供将来使用。

  • A类地址: 网络号占8位, 主机号占24位, 适用于超大型网络, 其中网络号的首位固定为0。
  • B类地址: 网络号占16位, 主机号占16位, 适用于中到大型网络, 其中网络号的前两位固定为10。
  • C类地址: 网络号占24位, 主机号占8位, 适用于小型网络, 其中网络号的前三位固定为110。
  • D类地址: 用于多播, 网络号占前4位, 其余位用于标识多播组, 其中网络号的前四位固定为1110。
  • E类地址: 保留地址, 供将来使用, 其中网络号的前四位固定为1111。

除此之外, 还要注意主机号全为0的地址是 网络地址，全为1的地址是 广播地址，都不能分配给具体设备。网络号127被保留用于 本地环回测试, 不能用于实际网络通信。

A类地址：

结构：8位网络号，24位主机号 。首位固定为0。

网络范围：1.0.0.0 至 126.0.0.0。

主机容量：每个网络可容纳约1677万台主机 (2 24 −2) 。

用途：为极少数超大型网络设计。

B类地址：

结构：16位网络号，16位主机号 。前两位固定为10。

网络范围：128.0.0.0 至 191.255.0.0。

主机容量：每个网络可容纳65,534台主机 (2 16 −2) 。

用途：为中到大型网络设计。

C类地址：

结构：24位网络号，8位主机号 。前三位固定为110。

网络范围：192.0.0.0 至 223.255.255.0。

主机容量：每个网络可容纳254台主机 (2 8 −2) 。

用途：为大量的小型网络设计。

重要规则：主机号全为0的地址是 网络地址，全为1的地址是 广播地址，都不能分配给具体设备 。网络号127被保留用于 本地环回测试 。

划分子网 (Subnetting)

假如一个C类网络 (例如192.168.1.0/24)，它的主机号部分有8位，去掉网络地址和广播地址后可用的主机地址是254个, 而现在有一个小型办公室只有10台主机, 这就浪费了244个地址; 另一个大型企业有1000台主机, 但C类网络最多只能容纳254台主机, 这就不够用了, 需要使用B类地址, 但B类地址又太大, 浪费了很多地址。

为了解决分类编址不够灵活、容易浪费地址的问题，引入了划分子网技术 。

核心思想是：从原有分类地址的主机号部分借用若干位作为子网号，将一个大的网络划分成多个更小的子网。这使得IP地址从二级结构（网络号+主机号）演变为三级结构（网络号+子网号+主机号）。

子网掩码 (Subnet Mask)：一个32位的数值，用于指明IP地址中哪些位是网络号和子网号，哪些位是主机号。子网掩码中连续的1对应网络号和子网号，连续的0对应主机号。我们只需要将一个IP地址与其子网掩码进行按位与（AND）运算，可以得到该IP地址所在的子网的网络地址 。

无分类编址 (CIDR)

划分子网的方法虽然提高了地址利用率，但仍然存在灵活性不足的问题。例如C类地址的主机号部分只有8位，最多只能划分出256个子网，而每个子网的主机数越划分越少，这仍然不够灵活。为此，引入了无分类编址 (Classless Inter-Domain Routing, CIDR)。

CIDR是目前因特网正在使用的编址方法，它消除了A、B、C类的概念，可以更有效地分配IP地址。

核心思想是：IP地址再次回归二级结构，但更加灵活：网络前缀 (Network-Prefix) + 主机号。网络前缀的长度是可变的。

一般采用的记法是CIDR记法（斜线记法）：在IP地址后面加上斜线/和网络前缀的位数来表示地址块。例如：128.14.35.7/20 表示前20位是网络前缀，后12位是主机号, 转换为二进制是10000000.00001110.00100011.00000111, 其中前20位 10000000.00001110.0010 是网络前缀，后12位 0011.00000111 是主机号。

使用CIDR的一个好处是可以更灵活地划分网络。例如，假设一个组织需要一个包含1000台主机的网络。使用CIDR，可以选择一个合适的前缀长度来满足这个需求。

另一个好处是CIDR支持路由聚合 (Route Aggregation)，也称为构造超网(Supernetting), 可以根据共同前缀将多个连续的网络地址块合并成一个更大的地址块，从而减少路由表的规模，提高路由效率。

IPv4地址的应用规划

IPv4地址的应用规划，指的是将一个给定的IPv4地址块（无论是传统的分类网络还是CIDR地址块），根据实际网络拓扑的需求，将其划分成若干个更小的地址块（即子网），然后将这些子网分配给互联网中的不同网络，并最终为各网络中的主机和路由器接口配置具体的IP地址的过程 。

这个规划过程主要有两种方法 ：

  • 采用定长的子网掩码 (Fixed Length Subnet Mask, FLSM)
  • 采用变长的子网掩码 (Variable Length Subnet Mask, VLSM)

定长的子网掩码 (FLSM)

在使用定长子网掩码（FLSM）进行子网划分时，所有划分出的子网都必须使用同一个子网掩码 。这意味着每个子网分配到的IP地址数量是完全相同的，这种“一刀切”的方式虽然简单，但非常容易造成IP地址资源的浪费。

假设我们申请到了一个C类网络 218.75.230.0，需要为下图所示的5个网络进行地址分配。

第一步：统计各网络地址需求: 我们需要计算每个网络总共需要多少个IP地址（包括主机、路由器接口、网络地址和广播地址）。

第二步：确定子网掩码: 确定所需子网数量：我们总共有5个独立的网络，需要划分出至少5个子网。我们需要从C类地址的8位主机号中借位来创建子网。如果借用n位作为子网号，则可以划分出2n 个子网。因此，必须借3位，这样可以划分出 2^3 =8 个子网，满足需求。

C类网络的默认掩码是 255.255.255.0 (即 /24)。从主机位借走3位后，新的子网掩码变为 /27 (24+3=27)。/27 对应的二进制掩码是 11111111.11111111.11111111.11100000, 转换成点分十进制就是 255.255.255.224。在FLSM中，所有5个网络都将使用这个掩码 。

第三步：分配子网: 借用3位后，每个子网的主机位还剩下 8−3=5 位，因此每个子网的大小为 2^5 =32 个地址。我们可以从划分出的8个子网中任选5个进行分配 。 - 子网1 (000): 218.75.230.0/27 (地址范围 .0 - .31) -> 分配给网络1 - 子网2 (001): 218.75.230.32/27 (地址范围 .32 - .63) -> 分配给网络4 - 子网3 (010): 218.75.230.64/27 (地址范围 .64 - .95) -> 分配给网络2 …以此类推

FLSM的缺点: 这种方法的弊端非常明显：地址浪费严重。以 网络5 为例，它实际只需要4个IP地址，但我们却不得不分配给它一个包含32个地址的子网，造成了28个地址的巨大浪费 。

变长的子网掩码 (VLSM)

变长子网掩码（VLSM）允许对同一个主网络划分出的不同子网使用不同的子网掩码。这样就可以根据每个子网的实际需求来精确分配地址数量，从而最大限度地减少地址浪费。

同样是上面的网络需求，但这次我们使用VLSM来从地址块 218.75.230.0/24 进行规划。

第一步：按需求排序并确定各子网所需的前缀长度

我们通常将需求按从大到小的顺序排列，以方便划分，避免地址碎片。

网络2 (需28个地址)：需要一个至少包含28个地址的块。2^5 =32，满足需求。主机位需要5位，因此网络前缀为 32 - 5 = 27，即 /27。

网络3 (需15个地址)：需要一个至少包含15个地址的块。2^4 =16，满足需求。主机位需要4位，因此网络前缀为 32 - 4 = 28，即 /28。

网络4 (需13个地址)：同样需要一个16个地址的块，前缀为 /28 。

网络1 (需9个地址)：同样需要一个16个地址的块，前缀为 /28。

网络5 (需4个地址)：需要一个至少包含4个地址的块。2^2 =4，满足需求。主机位需要2位，因此网络前缀为 32 - 2 = 30，即 /30 。

第二步：从大到小，依次划分地址块

我们从 218.75.230.0/24 这个大地址池中，像切蛋糕一样，一块一块地切分出去 。

  • 分配给网络2 (/27)：切出第一个块 218.75.230.0/27 (范围 .0 - .31), 变成二进制是 11011010.01001011.11100110.000xxxxx，其中前27位 11011010.01001011.11100110.000 是网络前缀，后5位 xxxxx 是主机号。
  • 分配给网络1 (/28)：从未分配的地址（从 .32 开始）中切出第二个块 218.75.230.32/28 (范围 .32 - .47)。变成二进制是 11011010.01001011.11100110.0010xxxx，其中前28位 11011010.01001011.11100110.0010 是网络前缀，后4位 xxxx 是主机号。
  • 分配给网络3 (/28)：继续切出第三个块 218.75.230.48/28 (范围 .48 - .63)。 变成二进制是 11011010.01001011.11100110.0011xxxx，其中前28位 11011010.01001011.11100110.0011 是网络前缀，后4位 xxxx 是主机号。
  • 分配给网络4 (/28)：继续切出第四个块 218.75.230.64/28 (范围 .64 - .79) 。
  • 分配给网络5 (/30)：最后切出第五个块 218.75.230.80/30 (范围 .80 - .83) 。

VLSM的优点: 通过这种方式，每个子网都得到了大小最合适的地址块，实现了按需分配。网络5 只需要4个地址，我们就精确地给了它一个包含4个地址的 /30 块，完全没有造成地址浪费 。剩余的172个地址（从.84到.255）可以留作未来扩展使用 ，极大地提高了IP地址的利用率。

IPv4地址与MAC地址对比

首先，我们需要理解这两种地址在TCP/IP协议体系结构中的位置。

IP地址工作在网络层, 它是一个逻辑地址，用于在整个因特网范围内唯一标识一台主机或路由器。在封装数据时，源IP地址 和 目的IP地址 被封装在 IP数据报的首部 中 。

MAC地址工作在网络接口层中的数据链路层。它是一个物理地址或硬件地址，固化在网络接口卡（网卡）上，用于在同一个局域网（或同一段链路）内标识一个具体的网络设备。在封装数据时，源MAC地址 和 目的MAC地址 被封装在帧的首部中。

如下图所示，应用层的数据逐层向下封装，IP地址在网络层添加，而MAC地址在更下一层的数据链路层添加。

并且, 当一个数据包从源主机H1发送到不同网络的目的主机H2，途中经过路由器R1和R2时，两种地址的变化规律是截然不同的。

IP地址：始终不变. 在整个传输过程中，IP数据报首部中的 源IP地址 (IP1) 和目的IP地址 (IP2) 始终保持不变 。它们标识的是通信的最初发起者和最终接收者。

MAC地址：逐跳改变. 数据链路层的帧首部中的源MAC地址和目的MAC地址在每一“跳”（hop）都会改变 。它们标识的是当前这一段链路上的发送者和接收者。

也就是说, IP地址负责“全程导航”（从起点到终点），而MAC地址负责“分段带路”（从当前一站到下一站）。

到这里, 可能一个常见的问题是：既然最终数据是在链路上靠MAC地址找到设备的，为什么不直接只用MAC地址通信，而要引入IP地址呢？结论是, 如果仅使用MAC地址进行通信，将会面临巨大的挑战 ：

  • 路由表爆炸：全世界每台路由器的路由表中，都必须记录因特网上所有主机和路由器接口的MAC地址 。这是一个天文数字，完全不具备可扩展性。
  • 管理极其困难：手工配置如此庞大的路由表是不可能的。即使使用路由协议自动构建，海量的MAC地址信息也会严重占用网络带宽和路由器的存储空间 。

而使用IP地址则完美地解决了这个问题。IP地址的层级结构（网络号+主机号）使得路由器只需记录到目标网络的路由，而无需记录网络内每一台主机的地址。这使得路由表的规模大大减小。

但是这就引出了另一个问题：路由器根据目的IP地址查路由表，知道了下一跳路由器的IP地址，但它不知道下一跳的MAC地址，数据链路层无法封装帧，怎么办？

这个“IP地址到MAC地址的最后一公里映射问题”，正是由地址解析协议（ARP）来解决的 。ARP负责在局域网内动态地查询IP地址对应的MAC地址，从而将网络层的IP寻址和数据链路层的MAC寻址无缝地衔接起来。

ARP 协议

ARP (Address Resolution Protocol)，即“地址解析协议”, 用于在一个局域网（LAN）内，将一个已知的IP地址（逻辑地址）解析（翻译）成对应的MAC地址（物理地址）。

当主机A想要给同一局域网内的主机B发送数据时，主机A通常只知道主机B的IP地址（例如，你在ping 192.168.1.101）。但是，数据在局域网内最终是以数据帧的形式传播的，而帧的头部必须包含目标MAC地址。交换机根据这个MAC地址来转发帧。因此，主机A在发送数据之前，必须先获得主机B的MAC地址。

ARP的工作原理

ARP的工作过程可以分为两个核心步骤：ARP请求 和 ARP应答。我们通过一个例子来说明。

假设局域网内有两台主机：

  • 主机A: IP地址 192.168.1.100，MAC地址 AA-AA-AA-AA-AA-AA
  • 主机B: IP地址 192.168.1.101，MAC地址 BB-BB-BB-BB-BB-BB

现在，主机A 想要向 主机B 发送数据。

步骤1：检查ARP缓存 主机A首先会检查自己的 ARP缓存表（ARP Cache）。这是一个存储了近期IP地址与MAC地址映射关系的动态表格(为了提高效率，避免每次通信都进行一次广播查询)。

如果找到 192.168.1.101 对应的MAC地址，则直接使用该地址封装数据帧并发送，整个ARP过程结束; 如果没有找到，则进入下一步。

步骤2：发送ARP请求（广播） 主机A会在局域网内发送一个 ARP请求（ARP Request） 报文。

这个请求的核心信息是：“谁的IP地址是 192.168.1.101？请告诉我的IP地址 192.168.1.100，我的MAC地址是 AA-AA-AA-AA-AA-AA”。

这个ARP请求报文会被封装在一个以太网帧中。这个帧的特殊之处在于源MAC地址是主机A的MAC地址 (AA-AA-AA-AA-AA-AA), 目标MAC地址却是一个特殊的广播地址 (FF-FF-FF-FF-FF-FF)。使用广播地址意味着这个帧会被发送到该局域网内的所有设备。

步骤3：网络中所有主机处理ARP请求 局域网内的所有设备（包括主机B和其他主机）都会接收到这个广播帧。

其他主机会检查ARP请求中的目标IP地址（192.168.1.101），发现与自身的IP地址不符，因此会静默丢弃这个请求，不作任何响应。

而主机发现ARP请求中的目标IP地址正是自己的IP地址, 执行下一个阶段。

步骤4：发送ARP应答（单播） 主机B在确认请求是发给自己的之后，会构造一个 ARP应答（ARP Reply） 报文。

报文内容：应答的核心信息是：“我就是 192.168.1.101，我的MAC地址是 BB-BB-BB-BB-BB-BB”。

这个ARP应答报文也会被封装在一个以太网帧中，但这次是单播：源MAC地址是主机B的MAC地址 (BB-BB-BB-BB-BB-BB), 目标MAC地址是主机A的MAC地址 (AA-AA-AA-AA-AA-AA)。主机B之所以知道主机A的MAC地址，是因为它在最初的ARP请求中已经包含了。

单播可以直接将应答信息发送给请求者（主机A），而不需要再打扰网络中的其他设备。

步骤5：更新ARP缓存并通信 主机A收到主机B的ARP应答后，就获取了 192.168.1.101 对应的MAC地址 BB-BB-BB-BB-BB-BB。

接着, 主机A会将这个映射关系存入自己的ARP缓存表，并设置一个老化时间（例如2分钟）。在老化时间内，再次向 192.168.1.101 发送数据时，就可以直接从缓存中查找，无需再次发送ARP请求。

主机A现在可以愉快地将数据封装成以太网帧（目标MAC为BB-BB-BB-BB-BB-BB），并通过交换机准确地发送给主机B了。

总之, ARP 协议可以认为是网络层在与数据链路层交互时所使用的一个“工具”或“接口”, 利用网络层的IP地址获取数据链路层的MAC地址，以便在物理网络上进行通信

ARP高速缓存表

ARP缓存表中的记录分为两种类型 ：

  • 动态 (Dynamic)：通过ARP协议自动获取的记录。它们有一个生命周期（例如2分钟），到期后会自动删除，以适应网络中可能发生的变化（如某台主机的网卡更换导致MAC地址改变）。

  • 静态 (Static)：由网络管理员手动配置的记录，通常不会自动过期 。

ARP的范围限制

一个非常重要的概念是，ARP协议只能在同一个局域网（或同一个广播域）上使用，不能跨网络（即跨路由器）工作 。

当主机H1要给另一个网络的主机H2发送数据时，它实际上是使用ARP来查找它的 默认网关（路由器R1） 的MAC地址。数据包在跨越多个网络的每一段链路（例如 H1 -> R1, R1 -> R2, R2 -> H2）上，都会独立地使用一次ARP协议来解析下一跳的MAC地址 。

IP数据报的发送和转发过程

该过程包含两个主要部分：主机如何发送IP数据报，以及路由器如何转发IP数据报。

主机发送IP数据报: 主机发送IP数据报时，首先需要判断目的主机是在同一个网络还是在不同的网络，这决定了数据报的交付方式 。

  • 直接交付：当源主机和目的主机在同一个网络中时，它们之间可以直接通信。
    • 源主机会通过ARP协议获取目的主机的MAC地址，然后将IP数据报封装成帧直接发送给对方
  • 间接交付：当源主机和目的主机位于不同网络时，通信需要通过它们的默认网关（路由器）来中转。
    • 源主机会将IP数据报发送给自己的 默认网关，由网关来负责后续的转发。

至于如何判断目的主机是否在同一网络，源主机会通过一个简单的计算来判断: 将源主机的IP地址和子网掩码进行按位与运算，得到源主机的网络地址；然后将目的主机的IP地址和同样的子网掩码进行按位与运算，得到目的主机的网络地址。如果两个网络地址相同，则说明两台主机在同一个网络中；否则，它们在不同的网络中。

路由器转发IP数据报: 当路由器收到一个IP数据报后，如果数据报是有效的（首部无误码且生存时间未耗尽），它会基于数据报首部中的目的IP地址，在自己的路由表中进行查询 。

  • 如果查询到匹配的路由条目，就按照该条目的指示进行转发 。
  • 如果查询不到匹配的条目，路由器会丢弃该IP数据报，并向源主机发送一个ICMP差错报告报文

路由器查表转发过程: 假设主机A向主机D发送一个IP数据报，其源地址为 192.168.0.1，目的地址为 192.168.0.129 。该数据报通过间接交付被发送到路由器R。

首先, 路由器R收到数据报后，首先检查数据报的完整性和生存时间。如果数据报有效，R会提取目的IP地址 192.168.0.129 。然后，R会在自己的路由表中查找该目的地址的最佳匹配条目。

逐条匹配路由表：路由器R会用路由表中每条条目的地址掩码与数据报的目的地址进行按位与运算，并将结果与该条目的目的网络地址进行比较。

  • 匹配第一条：将 192.168.0.129 与第一条路由的掩码 255.255.255.128 (/25) 进行“与”运算，结果是 192.168.0.128 。这个结果与第一条的目的网络 192.168.0.0 不匹配 。
  • 匹配第二条：将 192.168.0.129 与第二条路由的掩码 255.255.255.128 (/25) 进行“与”运算，结果是 192.168.0.128 。这个结果与第二条的目的网络 192.168.0.128 匹配 。

执行转发：路由器R找到了匹配的第二条路由条目, 该条目的“下一跳”指示为“接口1直连” 。这意味着目的主机D就在路由器R的接口1所连接的网络上。路由器R会通过ARP协议获取主机D的MAC地址，然后将IP数据报封装成帧，从接口1直接交付给主机D 。

对广播数据报的处理: 路由器是隔离广播域的。如果路由器收到的是一个目的地址为广播地址（如 192.168.0.127 或 255.255.255.255）的IP数据报，它不会对这种数据报进行转发。这样做是为了防止广播数据报在整个因特网中泛滥，造成巨大的网络风暴和资源浪费。

IP数据报的首部格式

IP数据报的首部格式及其内容是实现IP协议所有功能的基础 。IPv4数据报的首部由一个 20字节的固定部分 和一个最大 40字节的可变部分 组成 。固定部分是每个IPv4数据报都必须包含的，而可变部分（可选字段）则用于增加一些特殊功能，但很少被使用 。

在TCP/IP标准中，数据格式常以32比特（4字节）为单位来描述，首部格式的每一行都代表4个字节

以下是IPv4数据报首部中各个字段的详细含义：

  1. 版本 (Version)

    • 长度：4比特 。
    • 功能：用来表示IP协议的版本 。通信双方使用的IP协议版本必须一致 。目前广泛使用的版本号为4，即IPv4 。

  2. 首部长度 (Header Length)、可选字段和填充

    • 首部长度：
      • 长度：4比特 。
      • 功能：该字段的取值以 4字节 为单位，用来表示整个IPv4数据报首部的长度 。
      • 取值范围：最小值为二进制0101（即十进制5），表示首部长度为 5×4=20 字节，此时首部只包含固定部分 。最大值为二进制1111（即十进制15），表示首部长度为 15×4=60 字节 。
    • 可选字段 (Options)：
      • 长度：从1字节到40字节不等 。
      • 功能：用来支持排错、测量以及安全等措施 。但由于它会增加路由器处理数据报的开销，因此很少被使用 。
    • 填充 (Padding)：
      • 功能：由于可选字段的长度可变，可能导致首部总长度不是4字节的整数倍。填充字段用全0来补足，以确保首部长度始终是4字节的整数倍 。

  3. 区分服务 (Differentiated Services)

    • 长度：8比特 。
    • 功能：用来获得更好的服务质量（QoS） 。这个字段在旧标准中被称为服务类型（ToS），但一直未被广泛使用 。一般情况下不使用该字段 。

  4. 总长度 (Total Length)

    • 长度：16比特 。
    • 功能：该字段的取值以 字节 为单位，用来表示整个IPv4数据报的长度（即首部长度 + 数据载荷长度） 。
    • 取值范围：最大值为 216 −1 = 65535 字节 。

  5. 标识 (Identification)、标志 (Flags) 和 片偏移 (Fragment Offset) 这三个字段共同用于IP数据报的 分片与重组 。当一个IP数据报的长度超过了链路的最大传送单元（MTU）时，就必须将其分割成更小的片段（分片）进行传输 。

    • 标识 (Identification)：
      • 长度：16比特 。
      • 功能：这是一个唯一的标识符。当一个数据报被分片后，所有这些分片都具有 相同的标识值，这样目的主机就能知道哪些分片属于同一个原始数据报 。
    • 标志 (Flags)：
      • 长度：3比特 。
      • 功能：
        • 最低位 (MF - More Fragments)：MF=1 表示后面还有分片；MF=0 表示这是最后一个分片 。
        • 中间位 (DF - Don’t Fragment)：DF=1 表示不允许分片；DF=0 表示允许分片 。
        • 最高位：保留位，必须为0 。
    • 片偏移 (Fragment Offset)：
      • 长度：13比特 。
      • 功能：该字段的取值以 8字节 为单位，用于指明当前分片的数据载荷部分相对于原始数据报数据载荷部分的起始位置 。

  6. 生存时间 (Time To Live, TTL)

    • 长度：8比特 。
    • 功能：该字段的目的是防止IP数据报在网络中因路由环路而无限循环 。
    • 工作机制：每经过一个路由器，该字段的值就 减1 。当TTL的值变为0时，路由器会丢弃该数据报，并向源主机发送一个ICMP时间超过的差错报告 。

  7. 协议 (Protocol)

    • 长度：8比特 。
    • 功能：用来指明该IP数据报的数据载荷部分封装的是哪一种上层协议的数据单元（PDU） 。
    • 常用协议值：TCP为6，UDP为17，ICMP为1，OSPF为89 。

  8. 首部检验和 (Header Checksum)

    • 长度：16比特 。
    • 功能：用于检测IPv4数据报的首部在传输过程中是否出现了差错。它不检验数据载荷部分，以减少计算量。
    • 工作机制：
      • 发送方：将首部划分为多个16比特的字，然后用反码算术运算求和，最后将结果的反码写入检验和字段 。
      • 接收方/路由器：将收到的首部所有16比特字（包括检验和字段本身）再次用反码算术运算求和。如果结果为全1（取反码后为0），则表示首部没有出错；否则表示出现差错，丢弃该数据报 。
      • 由于TTL字段在每经过一个路由器时都会改变，所以 每个路由器都必须重新计算首部检验和 。
    • 在IPv6中，取消了首部检验和字段，以简化路由器的处理流程，提高转发效率 。

  9. 源IP地址 (Source IP Address) 和 目的IP地址 (Destination IP Address)

    • 长度：均为32比特 。
    • 功能：分别用来填写发送该IP数据报的源主机的IP地址和接收该IP数据报的目的主机的IP地址 。
