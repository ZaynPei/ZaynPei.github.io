进程的基本概念 (Process Concept)

什么是进程 (Process)

一个进程是 “一个正在执行中的程序” 。它不仅仅是代码，还包含了程序当前的活动状态 。

一个进程在内存中通常包含以下几个部分 ：

  • 文本段 (Text Section)：存放程序代码 。
  • 数据段 (Data Section)：存放全局变量 。
  • 堆 (Heap)：用于动态分配的内存 。
  • 栈 (Stack)：存放函数参数、局部变量和返回地址 。
  • 程序计数器 (Program Counter)：指示下一条要执行的指令地址 。
  • 进程状态 (Process State)

进程状态 (Process State)

一个进程在其生命周期中会经历多种状态的转换 ：

  • 新建 (New)：进程正在被创建 。
  • 就绪 (Ready)：进程已准备好，等待被分配给CPU运行 。
  • 运行 (Running)：进程的指令正在被处理器执行 。
  • 等待 (Waiting/Blocked)：进程正在等待某个事件的发生，如 I/O 操作完成 。
  • 终止 (Terminated)：进程已完成执行 。

进程控制块 (Process Control Block, PCB)

进程控制块 (Process Control Block, PCB)：是操作系统进行进程切换和管理的基础, 操作系统为每一个进程都维护一个 PCB，它包含了与该进程相关的所有重要信息 ，例如：

  • 进程状态 。
  • 程序计数器和 CPU 寄存器的值 。
  • CPU 调度信息（如优先级） 。
  • 内存管理信息（如页表） 。
  • I/O 状态信息（如已分配的设备） 。

进程调度 (Process Scheduling)

进程调度是操作系统管理多个进程以有效利用 CPU 的机制 。

调度队列

调度队列 (Scheduling Queues)：操作系统使用队列来组织进程 。

  • 作业队列 (Job Queue)：系统中所有进程的集合 。
  • 就绪队列/CPU队列 (Ready Queue/CPU Queue)：所有在主内存中、准备好执行的进程的集合 。
  • 设备队列 (Device Queues)：等待某个 I/O 设备的进程集合 。

上图展示了操作系统中用于管理进程的两种主要类型的队列：就绪队列 (Ready Queue) 和 I/O 设备队列 (I/O Device Queues)。

每个矩形框（如 PCB7, PCB2）代表一个进程控制块 (Process Control Block, PCB)，它是操作系统用来存储一个进程所有信息的数据结构。

位于图顶部的“ready queue”就是就绪队列，也被称为 CPU 队列 (The CPU queue)。它包含所有已经准备就绪、可以立即在 CPU 上运行的进程。这些进程正在等待操作系统调度程序（Scheduler）为它们分配 CPU 时间。

  • queue header（队列头部）包含 head 和 tail 两个指针，分别指向队列的第一个和最后一个进程的 PCB。
  • 在这个例子中，PCB7 和 PCB2 正在就绪队列中排队，等待被 CPU 执行。PCB7 在 PCB2 的前面。

而当一个进程需要执行 I/O 操作时（例如从磁盘读取数据），它会进入等待状态，并被放入相应 I/O 设备的等待队列中，直到 I/O 操作完成。

图中也展示了多个独立的 I/O 设备队列，每个设备都有自己的队列。

  • 磁盘单元0 (disk unit 0)：它的队列中有三个进程 PCB3、PCB14 和 PCB6，说明这三个进程都在等待从该磁盘设备完成 I/O 操作。
  • 终端单元0 (terminal unit 0)：PCB5 正在等待该终端设备的 I/O 操作。
  • 磁带单元0和1 (mag tape unit 0/1)：这两个设备的队列是空的，表示当前没有进程在等待它们。

上图则是一个 “排队图” (A queueing-diagram)，它详细描绘了一个进程在操作系统中的生命周期和状态转换路径。

矩形框（灰色）代表队列或导致状态变化的事件; 圆形/椭圆形框（蓝色）代表系统资源（如 CPU, I/O）或事件的结果; 箭头表示进程在不同状态和队列之间的流转方向。

具体来看, 一个进程从进入系统到最终退出的完整流程，以及在此期间可能发生的各种情况有：

  1. 进入就绪队列 (Ready Queue)

    一个新创建的或者从等待状态恢复的进程首先会进入就绪队列。这里的进程已经准备好了一切，只等待被分配 CPU。

  2. 分配 CPU 并运行

    操作系统的调度程序会从就绪队列中选择一个进程，并将其调度到 CPU 上运行。

  3. 从 CPU 离开的几种可能情况

    • 情况一：发起 I/O 请求 (I/O request)

      • 如果正在运行的进程需要进行读写文件、访问网络等 I/O 操作，它会发起一个 I/O 请求。

      • 随后，该进程会从 CPU 释放，并进入相应的 I/O 队列 (I/O queue) 进行等待。

      • 当 I/O 操作完成后，进程会重新回到就绪队列的末尾，等待下一次 CPU 调度。

    • 情况二：时间片用完 (Time slice expired)

      • 在分时操作系统中，每个进程只能在 CPU 上运行一个固定的时间段，即“时间片”。

      • 当时间片用完后，操作系统会通过一个中断来剥夺该进程的 CPU 使用权。

      • 该进程会直接回到就绪队列的末尾，等待下一轮调度。

    • 情况三：创建子进程 (Fork a child)

      • 进程在运行时可以调用 fork 之类的系统调用来创建一个新的子进程。

      • 创建子进程后，父进程可能会等待子进程执行完毕，或者继续并发执行。图中箭头指向父进程回到就绪队列。

    • 情况四：等待中断 (Wait for an interrupt)

      • 进程可能需要主动等待某个特定的事件发生，例如等待另一个进程发来的信号。

      • 在这种情况下，进程会进入等待状态，直到所等待的中断发生 (interrupt occurs)。

      • 中断发生后，该进程会被唤醒并移回到就绪队列中。

    • 情况五：正常结束

      • 进程执行完所有任务后会正常终止，然后退出系统。图中从 CPU 指向右侧外部的箭头就代表了这一路径。

调度程序 (Schedulers)

从概念上来看, 操作系统中的调度程序负责管理进程的执行, 可以有两种类型的调度程序：

短期调度程序 (Short-term Scheduler): 也称为 CPU 调度程序 (CPU scheduler) 。

它的主要任务是从就绪队列 (ready queue)中选择一个进程，并为它分配 CPU 。它决定了就绪队列中的哪个进程可以“上CPU”运行。

它的执行频率非常高，通常是毫秒级别，因为每次 CPU 需要切换进程时都可能调用它 。

由于执行得非常频繁，短期调度程序本身必须运行得非常快，否则会占用大量本该用于执行用户进程的时间 。

长期调度程序 (Long-term Scheduler): 也称为 作业调度程序 (job scheduler) 。

它的任务是从一个更大的进程池（例如硬盘上的所有待处理的“作业”）中，选择哪些进程应该被加载到内存中，并放入就绪队列 。

它控制着系统中的多道程序设计度 (degree of multiprogramming)，即同时存在于内存中的进程数量 。

一般来说, 它的执行频率非常低，可能几秒钟甚至几分钟才执行一次 。由于不经常执行，它的速度也相对较慢 。

值得注意的是，许多现代操作系统（如 UNIX 和 Windows）实际上并不使用长期调度程序 。

此外, 有些系统还引入了中期调度程序 (Medium-term Scheduler)，当内存不足或系统需要调整内存中的进程组合时，中期调度程序会选择一个进程（通常是处于等待状态或优先级较低的进程），将其从内存中暂时移除，并保存到硬盘等二级存储中。

在未来的某个时刻，当内存条件允许或该进程被需要时，中期调度程序会将被换出的进程重新加载回内存的就绪队列中，使其可以继续执行

在内存中, 进程大致可以分为计算密集型（CPU-bound）和I/O密集型（I/O-bound）进程, 前者主要进行大量CPU计算, 而后者则频繁进行I/O操作。

操作系统的调度程序通常会优先选择 I/O 密集型进程，因为它们在等待 I/O 操作完成时，CPU 可以被其他进程使用，从而提高整体系统的吞吐量和响应时间 。

上下文切换 (Context Switch)

当 CPU 从一个进程切换到另一个进程时，系统必须保存当前进程的状态（上下文）到其 PCB 中，并加载新进程的上下文 。

这个过程是纯粹的开销 (overhead)，因为在切换期间系统没有执行任何有用的工作 。

进程操作 (Operations on Processes)

这部分描述了进程的创建和终止过程。

进程创建 (Process Creation)：

一个进程（父进程）可以创建新的进程（子进程），从而形成一个进程树 。

在 UNIX/Linux 系统中:

  • fork() 系统调用用于创建一个新的子进程，该子进程是父进程的一个副本 。

    • 在创建后, 父进程和子进程会从 fork() 调用返回, 但返回值不同: 父进程收到子进程的 PID, 而子进程收到0 。
    • 子进程继承了父进程的大部分属性（如环境变量、打开的文件描述符等），但有自己的独立地址空间 。

  • exec() 系统调用通常在 fork() 之后被子进程调用，用于将新的程序加载到自己的内存空间中，以替代从父进程复制来的内容 。

  • 父进程可以通过 wait() 系统调用来阻塞等待子进程的完成

进程终止 (Process Termination)

当一个进程完成其最后一个语句的执行后，它会调用 exit() 系统调用来请求操作系统删除它 。其所有占用的资源（内存、文件等）都会被操作系统回收 。

父进程也可以终止其子进程（例如，当子进程超出了资源限制时）。

如果父进程终止，其子进程可能会被级联终止 (cascading termination) ，或者在某些系统中成为孤儿进程 (orphaned)，并被 init 进程（PID 为 1）所收养。

进程间通信 (Interprocess Communication, IPC)

这部分探讨了进程之间如何协作和交换信息。

协作进程 (Cooperating Processes)：

能够影响或被其他进程影响的进程称为协作进程 。

协作的优势包括信息共享、加快计算速度、模块化和便利性 。

生产者-消费者问题 (Producer-Consumer Problem)

生产者-消费者模型 描述了两个或多个并发实体（线程/进程）通过共享缓冲区（bounded buffer）协作完成任务的过程。

  • 生产者（Producer）：负责生产数据（例如消息、任务、商品）并放入缓冲区。
  • 消费者（Consumer）：从缓冲区取出数据进行处理或消费。
  • 缓冲区（Buffer）：是一个共享的、有界的存储空间，用来暂存生产者产生但消费者尚未取走的数据。

这个模型有两个关键挑战：

  • 同步（Synchronization）问题: 消费者不能在缓冲区为空时取数据, 生产者也不能在缓冲区满时放数据。因此需要一种方式让它们“有序等待、协调运行”。

  • 互斥（Mutual Exclusion）问题: 生产者和消费者不能同时访问缓冲区（否则会出现数据竞争）。因此需要互斥锁或信号量来保护共享资源。

IPC 的两种模型

共享内存 (Shared Memory)：多个进程共享一块内存区域。进程通过直接读写这块共享内存来交换信息。

消息传递 (Message Passing)：进程通过发送和接收消息进行通信，而无需共享变量 。这种通信通常由内核提供的 send() 和 receive() 操作来完成 。

消息传递的实现

直接通信 (Direct Communication)：进程必须明确地命名对方来进行通信，通信双方必须明确知道对方的身份（进程名或PID），通信链路由操作系统自动建立。 - send(P, message) → 向进程 P 发送消息
- receive(Q, message) → 从进程 Q 接收消息

间接通信 (Indirect Communication)：消息被发送到 邮箱 (mailbox) 或 端口 (port)，进程通过共享同一个邮箱来进行通信 。

同步与异步 (Synchronization)：通信可以是 阻塞的 (blocking/synchronous)，即发送方会等待消息被接收；也可以是 非阻塞的 (non-blocking/asynchronous)，即发送方发送消息后立即继续执行 。

客户端-服务器系统中的通信

这部分介绍了在网络环境中实现进程通信的几种常用技术。

套接字 (Sockets)：

套接字是通信的一个端点，它由一个 IP 地址和一个端口号组合而成 。两个进程间的网络通信就是通过一对套接字进行的 。

远程过程调用 (Remote Procedure Calls, RPC)：

RPC 是一种允许一个进程调用网络中另一台计算机上进程的过程（或函数）的机制，使得分布式编程看起来像本地调用一样简单 。

远程方法调用 (Remote Method Invocation, RMI)：

RMI 是 Java 中类似 RPC 的机制，它允许一个 Java 程序调用远程机器上另一个 Java 对象的

方法 。
