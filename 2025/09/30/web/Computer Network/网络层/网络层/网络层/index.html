网络层概述

网络层在OSI体系结构中是自下而上的第三层，其最核心的任务是将数据分组从源主机，经过多个网络和多段链路，最终传输到目的主机。这个核心任务可以被分解为两个关键功能：分组转发和路由选择 。

分组转发 (Forwarding)

当一台路由器从其某个接口所连接的链路上接收到一个数据分组后，它会根据分组首部中的信息，在自己的转发表中进行查询，并依据查询结果将该分组从一个合适的接口发送给下一跳路由器或目的主机 。

工作机制：

  • 每个路由器都维护一个转发表 (Forwarding Table) 。
  • 路由器依据分组首部中的转发标识（可能是目的地址或连接指示符）来查询转发表 。
  • 根据查询结果所指示的接口来执行转发操作 。

如图所示，路由器R1在其接口1收到了一个首部转发标识为”A”的分组。R1查询自己的转发表，发现标识”A”对应的出口是接口2，于是将该分组从接口2转发出去 。

路由选择 (Routing)

由于源主机和目的主机之间可能存在多条路径，网络层需要决定选择哪一条路径来传送分组，这个决策过程就是路由选择。路由选择方式主要有以下三种 ：

  • 集中式路由选择：由一个中央网络控制中心执行路由计算，并将路由信息下载给每个路由器 。
  • 分布式路由选择：每个路由器上都运行路由选择协议，通过相互交换路由信息来各自计算路由 。
  • 人工路由选择：由网络运维人员手动配置路由信息 。

路由表与转发表的关系：路由选择过程会生成路由表 (Routing Table)，它通常只包含从目的网络到下一跳的映射关系。而转发表是由路由表得出的，其结构被优化，以便于快速查找。为了简化讨论，在路由选择原理的探讨中，通常不严格区分二者

网络层向其上层提供的两种服务

网络层可以向其上层（传输层）提供两种截然不同的服务：面向连接的虚电路服务和无连接的数据报服务 。

  1. 面向连接的虚电路服务 (Virtual Circuit Service)

    • 核心思想：“可靠通信应由网络自身来保证” 。
    • 工作流程：
      • 建立连接：通信双方在开始通信前，首先需要建立一条网络层的逻辑连接，即虚电路 (Virtual Circuit, VC)，以确保通信所需的网络资源 。
      • 数据传输：所有分组都沿着这条已建立的虚电路按序进行存储转发传输。在连接建立后，每个分组的首部只需携带一个较短的虚电路编号，而无需完整的目的地址 。
      • 释放连接：通信结束后，需要释放之前建立的虚电路 。
    • 特点：这是一种面向连接的服务, 能够保证分组无差错、按序到达、不丢失、不重复; 分组沿着同一条逻辑路径进行转发, 如果路径上的某个节点出现故障，所有通过该节点的虚电路都将无法工作 。
    • 应用实例：曾经的X.25、帧中继(Frame Relay)和异步传输模式(ATM)等广域分组交换网使用此服务 。

  2. 无连接的数据报服务 (Datagram Service)

    • 核心思想：“可靠通信应当由用户主机来保证” 。

    • 工作流程：

      • 无需建立连接：网络层在通信前不需要建立任何连接 。
      • 数据传输：每个分组都独立选择路径，因此每个分组的首部都必须携带完整的目的主机地址 。
      • 无需释放连接：通信结束后，没有连接需要释放 。

    • 特点：这是一种无连接的服务 。网络本身不提供端到端的可靠传输保证，分组可能出现误码、丢失、重复和失序 。

因特网(Internet)的设计思想是将复杂的网络处理功能（如可靠性保证）放在网络边缘的用户主机上(而不是路由器)，而网络核心则提供相对简单的、尽最大努力（不可靠）的分组交付功能。这种设计使得网络中的路由器可以做得比较简单，造价低廉，运行方式灵活。因此，因特网采用的是无连接的数据报服务。

网络控制报文协议(ICMP)

网络控制报文协议 (Internet Control Message Protocol, ICMP) 是互联网协议族中的一个重要协议，主要用于在IP主机、路由器之间传递控制信息和差错报告。ICMP工作在网络层，通常被视为IP协议的一个组成部分。

ICMP的报文分为两大类：ICMP差错报告报文 和 ICMP询问报文 。

ICMP差错报告报文

ICMP差错报告报文用于向源主机或路由器报告在数据传输过程中遇到的差错情况。共有以下五种类型：

  1. 终点不可达 (Destination Unreachable)

    • 功能：当路由器或主机无法交付一个IP数据报时，就会向源主机发送“终点不可达”报文 。

    • 触发情景：例如，路由器R1收到一个发往网络N3的数据报，但在其路由表中找不到任何关于如何到达网络N3的路由条目（包括特定路由、网络路由或默认路由）。此时，R1会丢弃该数据报，并向源主机H1发送此差错报告 。

    • 细分类型：该报文还可以根据代码字段细分为目的网络不可达、目的主机不可达、目的协议不可达等13种具体错误 。

  2. 源点抑制 (Source Quench)

    • 功能：当路由器或主机由于网络拥塞而丢弃IP数据报时，会向源主机发送“源点抑制”报文，通知源主机应该降低其数据报的发送速率 。

    • 触发情景：

      • 路由器拥塞：数据报在传输途中经过的路由器R2因为过于繁忙而丢弃了该数据报 。
      • 主机拥塞：目的主机H2自身因为拥塞而丢弃了收到的数据报 。

  3. 时间超过 (Time Exceeded / Timeout)

    • 功能：用于通知源主机数据报因生存时间耗尽而被丢弃 。

    • 触发情景：

      • TTL为0：路由器收到一个IP数据报后，会将其首部中的生存时间（TTL）字段的值减1 。如果减1后结果为0，路由器必须丢弃该数据报，并向源主机发送“时间超过”报文 。
      • 分片重组超时：当目的主机在预设的时间内未能收到一个被分片的数据报的所有分片时，它会丢弃已收到的分片，并向源主机发送此报文 。

  4. 参数问题 (Parameter Problem)

    • 功能：当路由器或目的主机收到一个IP数据报，并发现其首部在传输过程中出现了误码时，会发送此报文 。

    • 触发情景：路由器R1在检查收到的IP数据报时，通过计算首部检验和发现首部字段有错误 。此时，R1会丢弃该数据报，并向源主机发送“参数问题”报文 。

  5. 改变路由 (Redirect)

    • 功能：路由器使用此报文来通知主机，有另一条更好的路径可以到达目的地址 。

    • 触发情景：主机H1的默认网关是R1 。当H1向网络N2发送数据报时，R1收到后发现，直接通过路由器R4到达N2的路径更优 。于是，R1会将数据报正常转发，但同时向H1发送一个“改变路由”报文，告知H1下次应直接将发往N2的数据报发送给R4 。

当然, 在某些情况下不应发送 ICMP差错报告报文，例如：对ICMP差错报告报文本身、对分片数据报的后续分片、对多播或特殊地址（如127.0.0.0）的数据报等 。

ICMP询问报文

ICMP询问报文用于主机或路由器之间的诊断和管理，帮助网络管理员了解网络状态和排除故障。常用的ICMP询问报文有两种 ：

  1. 回送请求和回答 (Echo Request and Reply)

    • 功能：由主机或路由器向一个特定的目的主机或路由器发出“回送请求”报文，收到请求的主机或路由器必须回复一个“回送回答”报文 。

    • 用途：这种询问报文主要用于测试目的站点是否可达以及了解其相关状态 。这是 PING 应用的基础。

  2. 时间戳请求和回答 (Timestamp Request and Reply)

    • 功能：用于请求某个主机或路由器回答当前的日期和时间 。
    • 用途：回答报文中包含一个从1900年1月1日起到当前时刻的总秒数，可被用来进行时钟同步和测量网络传输时间 。

ICMP的典型应用

ICMP协议虽然主要用于差错报告，但其询问报文机制也催生了两个非常著名且实用的网络诊断工具：分组网间探测（PING） 和 跟踪路由（traceroute/tracert）。

分组网间探测 (PING): PING（Packet InterNet Groper）是用于测试主机或路由器之间连通性的最基本工具。它能告诉我们目标设备是否在线（可达），以及数据包往返所需的时间。

工作原理：PING是一个应用层程序，但它不使用TCP或UDP，而是直接利用网络层的ICMP。其工作原理非常简单：

  • 源主机向目标主机发送一个 ICMP回送请求（Echo Request） 报文。
  • 目标主机在收到该请求后，必须回复一个 ICMP回送回答（Echo Reply） 报文。

注意：出于安全考虑，有些服务器或防火墙会设置不响应ICMP回送请求，这会导致PING不通，但这并不一定意味着目标主机不在线。

跟踪路由 (traceroute / tracert): traceroute（在Windows中为tracert）用于探测IP数据报从源主机到目的主机所经过的路由器路径。

UNIX/Linux (traceroute)：通常在运输层使用UDP协议，并利用ICMP的“终点不可达”差错报告报文。 Windows (tracert)：直接使用ICMP协议，它同时利用了ICMP的“回送请求/回答”报文和“时间超过”差错报告报文。

tracert 的巧妙之处在于它利用了IP数据报首部中的 TTL（生存时间） 字段和ICMP的 “时间超过” 报文。

假设主机H1要探测到H2的路径：

  • 探测第一跳：H1发送第一个IP数据报，其内部封装了一个ICMP回送请求，并将IP首部的 TTL设置为1。当这个数据报到达路径上的第一个路由器R1时，R1将TTL减1，变为0。由于TTL为0，R1会丢弃该数据报，并向源主机H1发送一个 ICMP时间超过的差错报告报文。H1收到这个来自R1的报文后，就知道了路径上的第一个路由器是R1。

  • 探测第二跳：H1发送第二个IP数据报，这次将 TTL设置为2。数据报到达R1，TTL减为1，被继续转发。数据报到达第二个路由器R2，R2将TTL减为0，丢弃数据报，并向H1发回一个 ICMP时间超过 报文。H1收到这个来自R2的报文后，就知道了路径上的第二个路由器是R2。

  • 依此类推：H1继续发送数据报，每次将TTL的值加1（TTL=3, 4, 5…），从而依次获得路径上第三个、第四个、第五个…路由器的IP地址。

  • 探测结束：当H1发送的某个数据报（例如TTL=99）的TTL值足够大，能够成功到达最终的目的主机H2时。H2收到的是一个ICMP回送请求报文，因此它不会发送“时间超过”报文，而是会回复一个 ICMP回送回答 报文。当源主机H1收到这个“回送回答”报文时，就知道已经到达了终点，跟踪过程结束。

虚拟专用网(VPN)和网络地址转换(NAT)

虚拟专用网(VPN)

VPN，全称为 虚拟专用网络 (Virtual Private Network)，是一种通过公共网络（通常是互联网）来建立安全、加密连接的技术。

我们可以通过一个比喻来理解它：想象一下，你要从家里寄一个非常重要的包裹到公司的总部。

  • 不使用VPN：你就像是把这个包裹直接交给了一个普通的快递公司。包裹在运输途中可能会经过很多个中转站，任何人都有机会看到包裹上的收件人信息，甚至可以偷偷打开看看里面的内容。
  • 使用VPN：这相当于你先把包裹放进一个坚固的、不透明的保险箱里（加密），然后再把这个保险箱交给一个专门的、只走秘密路线的安保运输队（隧道）。这个运输队会直接把保险箱送到公司总部的指定接收人手中。外界只能看到安保公司的车辆在公共道路上行驶，但完全不知道车里装的是什么，也不知道它的最终具体目的地是总部的哪个部门。

总结来说，VPN的核心作用就是在你和你的访问目标之间，建立一个临时的、安全的、私密的“信息隧道”。

VPN的核心工作原理

VPN的魔力主要来源于三大核心技术的结合：隧道技术、加密技术和身份验证。

  1. 隧道技术 (Tunneling)

这是VPN实现“虚拟专用”的基础。隧道技术指的是将原始的数据包（Packet）进行“再包装”或“封装”（Encapsulation）的过程。

  • 工作流程：

    • 你的设备（如电脑或手机）准备发送一个普通的数据包，上面有你的真实IP作为源地址，目标网站作为目的地址。

    • 在数据包离开你的设备之前，VPN客户端软件会拦截它。

    • VPN软件会将这个整个原始数据包（包括其头部信息）当作“货物”，塞进一个新的数据包（我们称之为“外部数据包”）的“集装箱”里。

    • 这个新的外部数据包的“信封”上，写的源地址是你自己，但目的地址不再是最终的网站，而是远程的VPN服务器。最终VPN服务器会代替你去访问真正的网站。

      • 如果是远程接入VPN (Remote Access VPN)，外部数据包的目的地址是VPN服务器的IP地址, 源地址是你的真实IP地址。
      • 如果是站点到站点VPN (Site-to-Site VPN)，外部数据包的目的地址是另一个局域网的网关路由器的IP地址, 源地址是你所在局域网的网关路由器的IP地址
  • 效果：你的互联网服务提供商（ISP）或者本地网络管理员，只能看到你正在和一台VPN服务器进行通信，但完全不知道你通过这台VPN服务器最终访问了什么网站。原始数据的真正目的地被隐藏在了这个“隧道”之中。
  1. 加密技术 (Encryption)

如果说隧道技术隐藏了你的目的地，那么加密技术就是保护你的数据内容本身。

工作流程：

  • 在将原始数据包封装进隧道之前，VPN会使用复杂的加密算法（如AES-256）和密钥 (Key) 将数据包的内容从可读的明文（Plaintext）转换成无法识别的乱码（Ciphertext）。

  • 这个被加密过的数据包随后被放入隧道，通过互联网传输到VPN服务器。

  • 只有拥有正确密钥的VPN服务器才能解密 (Decrypt) 这些乱码，将其恢复成原始数据。

  • 然后，VPN服务器会代替你，将解密后的原始数据包发送到你真正想访问的目标网站。

即使有黑客在传输过程中截获了你的数据，他们得到的也只是一堆毫无意义的乱码，无法窃取你的密码、银行信息或浏览内容。

  1. 身份验证 (Authentication)

为了确保你连接的是一个合法的VPN服务器，而不是某个黑客设置的假冒服务器，并且也为了让VPN服务器确认你是合法的付费用户，双方需要进行身份验证。

在你与VPN服务器建立安全隧道之前，你的VPN客户端和服务器会交换并验证彼此的数字证书或凭据（如用户名和密码）。这个过程确保了通信双方都是可信的，从而建立起一个安全的连接起点。

专用地址 (Private Address)

专用地址 (Private Address) 是指在专用网络（如企业内部网络或家庭网络）中使用的IP地址，这些地址在公共因特网中是不可路由的。这些IP地址仅在机构内部有效，可以由机构自行分配，无需向因特网管理机构申请。

地址范围：RFC 1918规定了三个地址块作为专用地址 ： - 10.0.0.0 ~ 10.255.255.255 (CIDR地址块 10/8) - 172.16.0.0 ~ 172.31.255.255 (CIDR地址块 172.16/12) - 192.168.0.0 ~ 192.168.255.255 (CIDR地址块 192.168/16)

因特网中的所有路由器都被配置为不转发目的地址是专用地址的IP数据报，从而保证了专用网络与公共因特网的隔离。

一次完整的VPN访问流程

让我们把所有原理串起来，看看当你连接VPN并访问网站时，发生了什么：

  • 启动连接：你在设备上打开VPN应用，选择一个服务器（例如，位于日本的服务器），然后点击“连接”。
  • 身份验证：你的设备与日本的VPN服务器进行身份验证，确认彼此的合法性。
  • 建立加密隧道：验证通过后，你的设备和VPN服务器之间建立起一个加密的通信隧道。
  • 发送请求：你在浏览器中输入 www.google.com。这个请求数据包被VPN软件加密，然后封装在一个新的数据包里，发往日本的VPN服务器。
  • 服务器中转：日本的VPN服务器收到数据包后，进行解密，看到了你真正的请求——访问谷歌。
  • 代替访问：VPN服务器以它自己（日本的IP地址）的身份，向谷歌服务器发送访问请求。
  • 谷歌响应：谷歌服务器将网站数据返回给日本的VPN服务器（在谷歌看来，访问者就来自日本）。
  • 安全返回：VPN服务器将收到的谷歌网站数据再次进行加密，通过安全隧道发回给你的设备。
  • 本地解密：你的VPN客户端收到加密数据后进行解密，最终在你的浏览器上显示出谷歌的网页。

整个过程中，你的所有网络流量都受到了端到端的保护，你的真实身份和位置也得到了完美的隐藏。

网络地址转换 (Network Address Translation, NAT)

网络地址转换（NAT）技术于1994年被提出，其主要目的是为了缓解IPv4地址空间即将耗尽的问题 。NAT允许大量使用内部专用地址的专用网络用户，通过共享少量外部全球地址来访问因特网 。

这项技术需要在连接专用网络和因特网的路由器上安装NAT软件，这种路由器被称为 NAT路由器 。NAT路由器至少拥有一个有效的外部全球IP地址 。当所有使用内部专用地址的主机与外部因特网通信时，它们的内部地址都必须在NAT路由器上被转换成这个全球IP地址 。

最基本的NAT方法

这是NAT最基础的工作方式，只转换IP地址。基本工作流程如下:

出站过程（从专用网到因特网）：

  1. 专用网内的主机A（使用专用地址 IP_A）向因特网上的主机B（使用全球地址 IP_B）发送一个IP数据报。该数据报的源地址是 IP_A，目的地址是 IP_B。

  2. 当NAT路由器收到这个数据报后，它会从自己的全球地址池中选择一个临时的全球地址 IP_G。

  3. 路由器将数据报首部中的源地址从 IP_A 修改为 IP_G。

  4. 同时，路由器会在自己的 NAT转换表 中记录下 IP_A 与 IP_G 的对应关系。

  5. 最后，将修改过源地址的数据报转发到因特网。

入站过程（从因特网到专用网）：

  1. 主机B回复一个IP数据报，其源地址是 IP_B，目的地址是NAT路由器为A选择的临时全球地址 IP_G。

  2. 当NAT路由器收到这个数据报时，它会查询NAT转换表。

  3. 根据表中的记录，NAT路由器知道这个数据报是发给主机A的。

  4. 路由器将数据报首部中的目的地址从 IP_G 修改回主机A的专用地址 IP_A，然后将其在专用网内转发给主机A。

这种最基本的NAT方法有一个限制：如果NAT路由器拥有n个全球IP地址，那么专用网内最多只能同时有n台主机接入因特网。

网络地址与端口号转换 (NAPT)

为了更有效地利用全球IP地址，现在普遍采用的是一种将NAT与传输层端口号结合使用的方法，称为 网络地址与端口号转换 (Network Address and Port Translation, NAPT) 。尽管技术上称为NAPT，但人们仍习惯将其简称为NAT 。

NAPT允许大量使用专用地址的内部主机，共同使用NAT路由器上的一个全球IP地址，同时与因特网上的不同主机进行通信。工作原理如下：

出站过程（内网到外网的NAPT转换）

  1. NAPT路由器收到内部主机（如192.168.0.6）发来的IP数据报 。

  2. 路由器不仅将其源IP地址（专用地址）修改为自己的外部全球地址（113.218.175.235），还会将其传输层PDU的源端口号（如30000）修改为一个由NAPT路由器动态分配的新的端口号（如50001） 。

  3. 路由器将这个包含 (内部地址:内部端口) -> (外部地址:外部端口) 的映射关系记录到自己的 NAPT转换表 中 。

  4. 最后，将修改过源地址和源端口号的IP数据报转发到因特网 。

入站过程（外网到内网的NAPT转换）

  1. 当NAPT路由器从因特网收到主机C发回的数据报时，它会检查数据报的目的地址和目的端口号 。

  2. 根据NAPT转换表中的记录（如 113.218.175.235:50001 对应 192.168.0.6:30000），路由器就能知道这个数据报应该发给专用网中的哪台主机的哪个应用 。

  3. 路由器将数据报首部中的目的地址和运输层PDU的目的端口号，按照NAPT转换表中的记录进行修改 。

  4. 最后，将数据报转发给内部的主机A 。

NAT/NAPT的特点与问题

  1. 通信必须由专用网内部发起 。

  2. NAT对网络应用并不完全透明，可能会对某些应用产生影响 。

  3. 拥有内部专用地址的主机不能直接充当因特网中的服务器 。

  4. 对于P2P这类需要外网主机主动与内网主机通信的应用，在通过NAT时会遇到问题，需要使用特殊的NAT穿透技术来解决 。

IP多播

随着因特网传输能力的提升和音视频压缩技术的发展，网上音视频业务（如视频点播、视频会议等）变得日益重要 。这些业务具有数据量大、对时延敏感的特点 。为了以最小的资源和时间来传输这些数据，需要采用不同于传统单播和广播的转发机制，IP多播技术因此应运而生 。

多播（Multicast），也称为组播，是一种实现“一对多”通信的技术。在因特网上进行的多播，则称为IP多播。与传统的“一对一”单播通信相比，多播可以极大地节省网络资源。

假如一个视频服务器需要向60个主机传送节目:

单播 (Unicast)：视频服务器需要为每一个接收视频的主机单独发送一份视频节目。如果要向60个主机传送，服务器就需要发送60个独立的视频节目副本，这会消耗大量的服务器和网络带宽资源。

多播 (Multicast)：视频服务器只需向一个特定的多播组发送 一个 视频节目副本即可。

数据流在网络中传输时，只有在路径的分叉点，路由器才会根据需要复制数据包。例如，路由器R1在转发时，会将视频节目复制成2个副本，分别发往R2和R3。

当数据包到达目的局域网时，可以利用局域网自身的硬件多播功能，无需复制即可让该网络上的所有多播组成员都收到这个视频节目。

因此, 当多播组的成员数量很大时，采用多播方式可以显著地减少网络中各种资源的消耗。要在因特网上实现多播，网络中的路由器必须具备多播功能，这涉及到IP多播数据报的寻址以及多播路由选择协议。

IP多播地址和多播组

IP多播通信必须依赖于IP多播地址。在IPv4中，D类地址 被指定为多播地址。

我们已经知道, D类IPv4地址的左起前4个比特固定为 1110, 剩余的28个比特可以任意变化，因此总共有 2^28 个IPv4多播地址。其点分十进制的范围是从 224.0.0.0 到 239.255.255.255 。

地址分类：根据[RFC3330]文档，D类地址可以进一步分为三类：

  • 预留的多播地址（永久多播地址）：例如 224.0.0.1 指代本子网上所有参加多播的主机和路由器，224.0.0.2 指代所有参加多播的路由器，224.0.0.5 用于OSPF路由器。

  • 全球范围可用的多播地址：可以在整个因特网范围内使用。

  • 本地管理的多播地址：仅在特定的本地范围内有效。

多播地址只能用作IP数据报的目的地址，不能用作源地址。每一个D类地址都用来标识一个多播组。所有使用同一个IP多播地址来接收多播数据报的主机，就构成了一个多播组 。

参与多播通信的主机，其 IP 地址始终是它本身在局域网内的单播 IP 地址（例如 192.168.1.100）。 多播地址（例如 239.1.1.1）仅用作数据包的目标地址，它代表的是一个“通信频道”或“群组”，而不是网络中任何一台具体的物理设备。

成员特性：

  • 多播组的成员是可以随时动态变动的，一台主机可以随时加入或离开多播组。
  • 多播组成员的数量和所在的地理位置不受限制 。
  • 一台主机可以同时属于几个不同的多播组 。

发送者特性：非多播组的成员也可以向任何一个多播组发送IP多播数据报 。

服务质量：IP多播数据报的传输遵循“尽最大努力交付”的原则，不保证一定能够成功交付给多播组内的所有成员。

在局域网上进行硬件多播

在因特网上进行IP多播的最后阶段，通常需要将多播数据报在本地局域网（如以太网）上交付给多播组的所有成员。幸运的是，局域网本身就支持硬件多播，即在数据链路层使用多播MAC地址进行“一对多”的帧传输。

为了利用这一功能，核心任务是将网络层的 IPv4多播地址（D类地址）映射成数据链路层的硬件多播地址（多播MAC地址）。

IP多播地址到MAC地址的映射规则

因特网号码指派管理局（IANA）专门为此保留了一段以太网MAC地址，范围是从 01-00-5E-00-00-00 到 01-00-5E-7F-FF-FF。 这些多播MAC地址的结构有一个特点：其左起前25个比特是固定的，只有剩余的23个比特可以任意变化。 因此，总共有 2^23 个以太网多播MAC地址。

映射规则是：将D类IP地址中可变的28个比特中的低23位，直接复制到多播MAC地址中可变的23位部分。

不过, 这个映射规则存在一个重要的问题：映射关系并不是唯一的。

因为一个D类IP地址有28个比特是可变的，而用于映射的多播MAC地址只有23个比特是可变的。 这意味着IP多播地址中，除了固定的1110外，前5个可变比特无法被映射到MAC地址中。

由于有5个比特的信息丢失了，多个不同的IP多播地址（最多 2^5 =32 个）可能会映射到同一个MAC地址。 #### 在网络层进行软件过滤

由于映射关系不唯一，仅仅依靠硬件（网卡）根据MAC地址来接收多播帧是不够的。因此，收到多播数据报的主机还需要在 网际层利用软件进行二次过滤。

  • 硬件（MAC层）过滤：当一个多播帧在局域网上传播时，所有加入了对应硬件多播组（拥有相同多播MAC地址）的主机网卡都会接收这个帧。 其他主机则在MAC层直接丢弃该帧。

  • 软件（网际层）过滤：接收了该帧的主机将帧解封装，把IP数据报向上传递给网际层。网际层会检查IP数据报首部中的目的IP地址。

    • 主机将这个目的IP地址与自己真正加入的IP多播组地址进行比较。如果地址不匹配，说明这个数据报虽然MAC地址正确，但并非发给自己的，于是将其丢弃。 只有当目的IP地址也匹配时，才会最终接受该数据报。

通过这种“硬件粗过滤 + 软件精过滤”的两层机制，即使存在地址映射冲突，也能保证只有正确的IP多播组成员才能最终收到并处理数据报。

IP多播需要的两种协议

要在因特网上实现IP多播，仅仅有IP多播地址是不够的。多播路由器必须能够智能地转发多播数据报，即只将数据报转发到那些确实有该多播组成员存在的网络中。 为了实现这一目标，需要两种协议协同工作：网络组管理协议 (IGMP) 和多播路由选择协议（PIM）。

网络组管理协议 (Internet Group Management Protocol, IGMP)

IGMP用于让连接在 本地局域网 上的多播路由器，知道本局域网上是否有主机（或主机中的某个进程）加入或退出了某个多播组。 它解决了“最后一公里”的成员关系问题。例如，在图4-112中，路由器R1和R2需要知道它们各自连接的局域网上有226.128.9.26这个多播组的成员，而R3需要知道它自己的局域网上没有该组的成员。 这个信息正是通过IGMP来获取的。

IGMP有三种主要的报文类型 ： - 成员报告报文 (Membership Report) - 成员查询报文 (Membership Query) - 离开组报文 (Leave Group)

IGMP报文被封装在IP数据报中进行传送, IP数据报首部中的协议字段的值为2，表示其数据载荷是IGMP报文。

由于IGMP仅在本网络(局域网)有效，封装有IGMP报文的IP多播数据报，其首部中 生存时间 (TTL) 字段的值被设置为1，以避免被路由器转发到其他网络 。

IGMP的基本工作原理

  1. 加入多播组

    • 主机操作：当一个主机希望加入某个多播组时，它会向其所在的网络发送一个封装有IGMP成员报告报文的IP多播数据报, IGMP成员报告报文中包含要加入的IP多播组的地址, 这个IP多播数据报的目的地址也设置为要加入的IP多播组的地址 。

    • 网络响应：该网络中所有该多播组的成员以及多播路由器都会收到这个报文 。如果本网络中还有同组的其他成员正准备发送加入报告，它们监听到这个报文后就会取消自己的发送，以减少网络冗余 。

    • 路由器操作：多播路由器会维护一个多播组列表，记录其直连网络中存在成员的多播组地址 。如果路由器收到一个未知多播组的成员报告，它就会将该多播组的地址添加到自己的列表中 。路由器只关心网络中有哪些多播组，而不关心具体有哪些成员。

  2. 监视多播组的成员变化

    • 路由器查询：多播路由器会周期性地（默认每隔125秒）向其直连网络发送一个IGMP成员查询报文, 该查询报文被封装在IP多播数据报中，目的地址为特殊的IP多播地址 224.0.0.1，这意味着本网络中所有参加多播的主机和路由器都会收到该报文 。

    • 主机响应：收到查询报文的多播组成员会发送一个IGMP成员报告报文作为应答。为了避免不必要的重复应答，主机采用 延迟响应策略：在1到10秒的随机时间内等待后再响应 。如果在此期间监听到同组其他成员已发送报告，则取消自己的响应 。

    • 路由器更新列表：如果多播路由器长时间没有收到某个特定多播组的成员报告，它就会将该多播组从自己的多播组列表中删除，认为本网络中已没有该组的成员 。

    • 在一个网络中若有多个多播路由器，它们会选举出一个查询路由器（通常是IP地址最小的那个），由它来负责发送成员查询报文，以避免重复查询 。

  3. 退出多播组

    • 主机操作：当一个主机要退出某个多播组时，它可以主动发送一个离开组报文，而无需等待路由器的查询 。

    • 离开组报文被封装在IP多播数据报中，目的地址为特殊多播地址 224.0.0.2，这个地址指向本网络中的所有多播路由器 。

    • 路由器操作：收到离开组报文后，路由器不能立即删除该多播组，因为网络中可能还有该组的其他成员。路由器会立即针对该特定多播组发送一个IGMP成员查询报文，以确认是否还有其他成员存在 。

      • 如果在预定时间内没有收到任何该组的成员报告报文，路由器才会将该多播组从自己的多播组列表中删除 。

正因为IGMP 仅在本网络中有效。它不能告诉路由器多播组包含了多少成员，也不能知道这些成员都分布在因特网的哪些网络中。因此，仅使用IGMP并不能在整个因特网上进行多播。要实现这一目标，还需要多播路由选择协议。

多播路由选择协议 (Multicast Routing Protocol)

多播路由选择协议用于在因特网上的多播路由器之间协同工作，以便用最小的代价将多播数据报传送给所有的组成员。

其主要任务是为每一个多播组建立一个多播转发树 (multicast forwarding tree)。这个转发树连接了多播源和所有拥有该多播组成员的路由器。 IP多播数据报只要沿着这个已经建好的转发树进行洪泛，就能被准确地传送到所有需要它的多播路由器。之后，在多播路由器所直连的局域网内，路由器再通过硬件多播将IP多播数据报发送给所有该多播组的成员。

需要注意的是, 为了覆盖所有的多播组成员, 多播转发树通常会包含一些没有该多播组成员的路由器和网络, 这是因为这些路由器和网络位于源和其他多播组成员之间的路径上, 它们是连接源和其他多播组成员的桥梁

目前有两种主流的方法来构建多播转发树：

  • 基于源树 (Source-Based Tree) 多播路由选择
  • 组共享树 (Group-Shared Tree) 多播路由选择
基于源树的多播路由选择

这种方法最典型的算法是 反向路径多播 (Reverse Path Multicasting, RPM) 。RPM算法包含两个关键步骤：

步骤一：利用反向路径广播 (RPB) 建立广播转发树

首先创建一个覆盖所有路由器的无环路的广播转发树，以避免广播分组在有环路网络中无限兜圈。

当一个路由器收到一个广播分组时，它会先检查该分组是否是从源点沿着最短路径传送来的。 - 检查方法：判断发送该分组的邻居路由器，是否就是自己到源点的最短路径上的第一个路由器 - 转发决策： - 是：如果路径正确，则将该分组从除了接收接口之外的所有其他接口转发出去 。 - 否：如果路径不正确（即分组来自一个非最短路径），则丢弃该分组，不再转发 。

通过这个机制，可以确保每个路由器只接收并转发一份来自最短路径的广播分组，从而形成一个以源点为根的、无环路的广播转发树 。

步骤二：利用剪枝 (Pruning) 算法获得多播转发树

RPB建立的是一个广播树，会把数据发送给所有路由器 。为了实现多播，需要从这个广播树上“剪掉”那些不需要接收数据的分支。

如果多播转发树上的某个路由器（叶节点）发现自己所连接的网络中没有该多播组的成员，并且它也没有下游路由器，它就会向上游路由器发送一个** 剪枝报文** (Pruning message) 。

上游路由器收到剪枝报文后，就会将通往该下游路由器的分支从转发树上剪除 。

嫁接 (Grafting)：如果之后被剪枝的路由器又通过IGMP发现了新的多播组成员，它可以向上游路由器发送一个 嫁接报文 (Grafting message)，重新加入到多播转发树中 。

组共享树多播路由选择

这种方法不为每个源都建立一棵树，而是让一个多播组的所有成员共享一棵树。

核心思想：在每个多播组中指定一个 核心 (core) 路由器 。以这个核心路由器为根，建立一棵连接该多播组所有成员路由器的生成树，作为多播转发树 。

建树过程：

  • 当一个成员路由器想要加入多播组时，它会向该组的核心路由器单播一个 加入报文 。

  • 这个加入报文会沿着单播路径朝核心路由器前进，直到它遇到核心路由器或某个已经属于该多播转发树的节点 。

  • 这个加入报文所经过的路径，就成为一个新的分支，被“嫁接”到现有的多播转发树上 。

源主机发送数据：如果一个不属于该多播组的源主机要发送多播数据，它会将多播分组封装在单播分组中，利用 IP隧道技术 发送给核心路由器 。核心路由器收到后再解封，并在多播转发树上进行洪泛 。

不过, 目前还没有一个在整个因特网范围统一使用的多播路由选择协议。一些被建议使用的协议包括：

  • 距离向量多播路由选择协议 (DVMRP)
  • 开放最短路径优先的多播扩展 (MOSPF)
  • 协议无关多播-稀疏方式 (PIM-SM)
  • 协议无关多播-密集方式 (PIM-DM)
  • 基于核心的转发树 (CBT)

而且, 至今IP多播在因特网上仍未得到大规模应用，主要应用在一些局部的园区网或专用网络中。目前，P2P技术的广泛应用推动了应用层多播技术的发展 。

总之, 这两种协议的角色分工明确且互为补充。IGMP 负责 “自下而上” 地向本地路由器报告成员情况，而 多播路由选择协议 则负责在路由器之间 “横向” 地构建高效的数据分发路径（多播转发树）。只有将两者结合，才能在庞大的因特网上实现高效、准确的IP多播。

移动IP技术

所谓移动IP (Mobile IP) 技术，是指允许移动设备（如智能手机、笔记本电脑等）在不同的网络之间切换时，仍然能够保持其IP地址不变，从而实现无缝的网络连接和通信。

例如这样一个场景: 用户在行驶的汽车中，设备从一个Wi-Fi网络无缝切换到另一个Wi-Fi网络，同时希望能不中断文件下载任务 。

核心矛盾：

  • 必须改变IP地址：当设备进入一个新的网络时，它的IP地址必须改变，否则数据包将无法被正确路由到这个新位置 。路由器会把发往其旧IP地址的数据包都路由到原来的网络 。

  • 不能改变IP地址：如果IP地址改变了，上层建立的TCP连接（如下载任务）会因为地址变化而失效，导致通信中断 。

移动IP技术正是为了解决这个“既要改变地址以实现路由，又不能改变地址以维持连接”的矛盾而设计的。它使得移动主机在不同网络间漫游时，能够保持其原来的IP地址不变。它通过引入一套新的角色和地址来实现这一目标，并且设计上无需改变非移动主机或大多数路由器的软件。

几个关键概念：

  1. 归属网络、归属地址和归属代理

归属网络 (Home Network)：每个移动主机都有一个默认连接或初始申请接入的网络。

归属地址 (Home Address)：移动主机在归属网络中拥有的一个永久IP地址，在整个移动通信过程中始终不变。

归属代理 (Home Agent, HA)：位于归属网络中的一个实体（通常是路由器），负责执行移动管理功能 。当移动主机离开归属网络时，归属代理会为它代收所有数据报 。

  1. 外地网络、外地代理和转交地址

外地网络 (Foreign Network)：移动主机当前漫游所在的网络，也称被访网络。

外地代理 (Foreign Agent, FA)：位于外地网络中的一个实体（通常是路由器），帮助移动主机执行移动管理功能。

转交地址 (Care-of Address, CoA)：由外地代理提供给移动主机的一个临时IP地址，该地址属于外地网络, 用于标识移动主机在外地网络中的位置。

移动IP技术的基本工作原理

移动IP的整个工作过程可以分为几个步骤：

  1. 代理发现与注册

    • 移动主机A从其归属网络漫游到一个外地网络。

    • 移动主机A通过“代理发现协议”与外地网络中的外地代理建立联系，并从外地代理处获得一个临时的转交地址（例如 175.1.1.1/16）。

    • 移动主机A（或通过外地代理）向其归属网络中的归属代理进行注册，告知自己的永久地址和当前的转交地址。

    • 归属代理记录下移动主机的转交地址，此后开始代替移动主机接收所有发往其永久地址的数据报。

  2. 固定主机向移动主机发送IP数据报（间接路由）

    • 发送到归属网络：固定主机B像往常一样，将IP数据报发送到移动主机A的永久地址 (218.75.230.16) 。数据报被正常路由到A的归属网络 。

    • 隧道转发：归属代理截获这个数据报，然后使用IP隧道技术（也称IP-in-IP封装）将其重新封装在一个新的IP数据报中 。这个新数据报的目的地址是移动主机A的转交地址 (175.1.1.1) 。这个过程对固定主机B是完全透明的 。

    • 交付移动主机：封装后的数据报通过因特网被路由到外地代理 。外地代理收到后，解封，取出原始的IP数据报，然后将其在外地网络中直接转发给移动主机A 。

  3. 移动主机向固定主机发送IP数据报（直接路由）

    • 移动主机A向固定主机B发送数据时，无需绕道归属代理 。

    • 它可以直接将IP数据报发送出去，源地址仍然填写其永久地址 。

    • 由于路由器转发数据报时只关心目的地址，不关心源地址，因此这个数据报会被直接路由到固定主机B 。

  4. 三角形路由问题 (Triangle Routing Problem)

不过, 容易看出上述通信路径是不对称的。从固定主机到移动主机的数据报必须先绕道归属网络，再通过隧道转发，形成一个“三角形”的低效路径 。

即使固定主机B和移动主机A位于同一个外地网络，B发给A的数据包仍然要先发送到遥远的归属网络，再被转发回来，造成极大的延迟和资源浪费 。

解决方案：一种方法是让固定主机也配置一个通信代理，该代理先从归属代理获取移动主机的最新转交地址，然后直接通过隧道将数据发送过去。但这会增加复杂性，并破坏了对固定主机透明的原则 。
